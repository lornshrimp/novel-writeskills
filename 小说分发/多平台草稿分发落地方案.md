## Plan: 多平台草稿分发落地方案

在不重写现有“多平台内容生成与门禁”流程的前提下，新增一层“上传分发层”：以现有 `多平台输出` 日志中的 `outputPath + gates` 作为唯一上游输入，优先接入能稳定落草稿的平台（微信公众号 API、小红书复用成熟发布后端），再逐步扩展到浏览器自动化平台。推荐采用“内容层与分发层解耦、平台适配器隔离、人工接管保底、批量任务可恢复”的路线，先做 3 平台 MVP，再扩容。

**Steps**
1. **冻结边界与复用原则**：保持现有 `.github/prompts/多平台输出SOP.prompt.md`、`scripts/platform_validate.ps1`、`scripts/scan_text_quality.ps1`、相似度门禁与 `SOP执行日志/多平台输出/` 结构不变；新增“上传分发层”只消费已通过门禁的本地平台稿。*这是后续所有步骤的前提。*
2. **定义统一上传任务模型**：约定单章上传任务的输入字段至少包含 `chapterNo`、`platform`、`outputPath`、`title`、`gatesPassed`、`sourceLogPath`、`attempt`、`idempotencyKey`、`targetWorkId`。其中 `idempotencyKey` 推荐使用 `chapterNo + platform + output file hash`，用于避免重复建草稿。*依赖 1。*
3. **新增分发配置层**：建立两类配置文件：
   - 平台目标配置：记录每个平台对应的作品/专栏/书籍 ID、章节命名规则、草稿模式、内容字段映射；
   - 认证配置：记录平台登录方式（API Token / OAuth / 浏览器 profile / storage state / cookies），并把密钥留在环境变量或忽略文件中。*依赖 2。*
4. **新增分发日志与恢复机制**：建立独立目录（建议 `SOP执行日志/多平台分发/`），日志结构沿用现有 SOP 思路，至少记录 `status`、`platformResults[]`、`remoteDraftId/url`、`error`、`lastCheckpoint`、`retryCount`。要求支持从单平台失败点继续，而不是整批重跑。*依赖 2，可与 3 并行。*
5. **抽象平台适配器接口**：设计统一适配器接口，至少包括：`checkAuth()`、`prepareDraftPayload()`、`createOrUpdateDraft()`、`verifyDraft()`、`captureEvidence()`。平台实现分为三类：
   - API 型（优先）：微信公众号；
   - 外部后端/现成技能复用型：小红书；
   - 浏览器自动化型：知乎、起点、番茄、豆瓣、B站、微博、百家号、WebNovel、My Fiction、GoodNovel、出版社后台。*依赖 3、4。*
6. **第一阶段 MVP（推荐先做 3 平台）**：
   - **微信公众号**：优先走官方草稿箱 API / API 封装项目；
   - **小红书**：优先复用 `BetaStreetOmnis/xhs_ai_publisher` 或 `Youhai020616/xiaohongshu` 的登录态与发布能力，不自写第一版底层；
   - **知乎或微博二选一**：若你更在意“小说章节长文”，先做知乎；若更在意低风险验证自动分发闭环，先做微博。*依赖 5。*
7. **MVP 上传链路编排**：实现“读取最近一次 `多平台输出` 成功日志 → 过滤 gates 全通过的平台稿 → 生成上传任务 → 逐平台上传 → 记录回执与截图证据 → 汇总摘要”的单章流程。批量章节在 MVP 阶段采用“章节内串行、章节间可排队”的保守模式。*依赖 6。*
8. **人工接管点设计**：以下情况必须暂停自动推进并转人工：登录失效、验证码/滑块、目标作品 ID 无法匹配、平台返回敏感词/风控提示、上传后校验到标题或字数异常。接管后继续写同一条分发日志，而不是另起任务。*依赖 5，可与 7 并行细化。*
9. **第二阶段扩容（浏览器自动化平台）**：优先级建议：
   - 第二批：起点、番茄、知乎、豆瓣、微博、百家号；
   - 第三批：B站、WebNovel、My Fiction、GoodNovel、出版社后台。
   浏览器自动化统一采用“真实 Chrome 持久化 profile + Playwright/CDP + 平台选择器适配层”的路线，避免为每个平台发明不同调度器。*依赖 7、8。*
10. **验证与运维机制**：为每个平台建立最小验证闭环：`auth check → dry run / preview → save draft → read back draft id/title/length`。再增加健康检查与小规模冒烟：每次平台页面改版或登录态过期后，先做 1 章试投，再恢复批量。*依赖 7。*
11. **范围控制与明确不做**：
   - 本方案只覆盖“将指定章节上传到对应平台并保存为草稿”；
   - 不纳入自动正式发布；
   - 不在第一阶段追求 13 平台同天上线；
   - 不把内容生成、平台风格改写、门禁逻辑搬到分发层重做。*贯穿全程。*

**Decisions**
- 推荐技术路线：`现有内容层保持不动 + 新增上传分发层 + 平台适配器隔离 + 人工接管保底`。
- 推荐首批平台：微信公众号、小红书、知乎或微博（二选一，取决于你更想先验证长文草稿还是低风险闭环）。
- 推荐外部复用策略：小红书优先复用成熟项目，不在第一版手搓登录态与发布流程。
- 推荐执行节奏：先 3 平台 MVP，再扩 6-8 平台，最后碰高异质平台（出版社后台、国际平台）。
- 明确排除：自动正式发布、13 平台一次性全量上线、在分发层重做内容门禁。

**Further Considerations**
1. **首批第三个平台如何选**：
   - 选知乎：更接近你的长文小说场景，但浏览器/编辑器复杂度更高；
   - 选微博：更容易验证“任务编排 + 账号认证 + 平台回执”闭环；
   - 推荐：如果你要尽快看到 MVP 成功，先微博；如果你要验证真正的“长文章节草稿”，先知乎。
2. **平台凭据管理方式**：
   - 方案 A：全部环境变量 + 本地 `.env` / 凭据文件；
   - 方案 B：浏览器 profile / storage state 文件集中管理；
   - 推荐：API 平台用 A，浏览器自动化平台用 B，混用但不要混存。
3. **MVP 是否直接引入外部项目**：
   - 方案 A：把外部项目作为独立服务调用，改造少；
   - 方案 B：直接抽取其中脚本进本仓库，控制力更强；
   - 推荐：MVP 用 A，稳定后再决定是否内化。
