# 短篇小说集最佳实践：从灵感到多平台分发（题材 Prompt 入口 + 门禁驱动版）

> 适用范围：本仓库所有题材目录（如 `女频爱情/`、`都市悬疑/`、`异能志怪/` 等）
>
> 核心原则：**用户直接调用题材目录下的 Prompt（`/斜杠命令`），并通过 `#引用` 提供上下文。**

---

## 1. 先统一调用架构（短篇集也不例外）

统一按这条链路工作：

`用户任务 -> /题材命令 -> 题材 Skill 路由 -> 通用 Skill 能力本体`

意味着：

- 用户侧入口是 `/命令`（例如 `/竞对分析`、`/设计大纲`、`/创建小说正文`）。
- 上下文通过 `#文件` 或 `#目录` 引用（例如 `#小说总大纲.md`）。
- 不在对话里写“执行 `女频爱情/.github/prompts/xxx.prompt.md`”。

---

## 2. 短篇小说集与长篇的关键差异

短篇小说集的核心难点，不是“单篇写好”，而是“**单篇可独立 + 全集有统一识别**”。

需要同时满足两层：

1. **单篇层**：每篇都能独立成立（冲突、回报、结尾余韵完整）。
2. **全集层**：主题、情绪谱系、题材承诺、风格边界有统一感。

一句话：**短篇集不是散装文件夹，而是有编排逻辑的作品系统。**

---

## 3. 先回答一个关键争议：集合主题未成型，能否先做篇目卡/人设/设定？

**结论：可以做轻量版本，但不应该做重定稿版本。**

### 3.1 为什么不能过早重投入

- 全集主题、篇间关系、阅读动线在早期都会变化。
- 如果过早重写篇目卡详细描述、人物传记、设定文档，后续大纲调整时会出现高返工、口径冲突、篇序失配。
- 短篇集的难点在于"统一辨识度"，而不是"堆砌内容"——建模过头反而会绑手脚。

### 3.2 最优做法：两段式建模

1. **全集主题前（轻量底盘）**：只保留最小可行信息
   - 篇目池：一句话冲突、独特回报、结尾策略（3~5 行）
   - 人物：若涉及跨篇人物，仅记录核心欲望 + 主要冲突（非完整传记）
   - 设定：若涉及跨篇规则，仅记录 3~5 条硬边界（非详细拆解）
   
2. **全集主题+篇序定稿后（重度定稿）**：才做大规模建模
   - 扩写完整篇目卡（涉及中段回报、文化细节、后期反转）
   - 补全人物传记（关系网、心理弧线、跨篇成长）
   - 完善技术设定（失效条件、证据链、程序摩擦）

## 4. 最优流程：6 段 5 门禁（短篇集版）

1. 立项段：题材定位 + 竞对
2. 选题段：短篇池设计（题目卡）
3. 架构段：全集总纲 + 单篇清单
4. 生产段：逐篇创作 + 逐篇审阅回炉
5. 编排段：篇目排序 + 全集一致性修订
6. 发行段：投稿组包 + 多平台分发 + 数据回流

### 门禁 S1（立项 -> 选题）

- 集合主题一句话可复述
- 目标平台竞对差距清晰
- 明确“读者为什么要连续读完多篇”

### 门禁 S2（选题 -> 架构）

- 至少形成 8~20 篇候选短篇题目卡（按计划体量调整）
- 每篇题目卡有：主冲突、独特回报、结尾钩子
- 题目之间无高重复

### 门禁 S3（架构 -> 生产）

- 有全集总纲（主题、风格、禁行项、篇间关系）
- 有单篇优先级队列（先写哪几篇）
- 每篇输入依赖明确（人物/设定/线索边界）

### 门禁 S4（生产 -> 编排）

- 每篇至少过一轮“审阅 -> 回炉”
- 单篇达成：开头抓力、核心回报、结尾余韵
- 文件命名、标签、元数据完整

### 门禁 S5（编排 -> 发行）

- 篇目顺序有明确阅读动线（起势 -> 变化 -> 收束）
- 全集语气与价值承诺一致
- 投稿版与平台版切换规则明确

## 6. 调用模板与最小输入清单（/命令 + #引用）

### 6.1 通用调用模板（可直接复制）

> /【命令名】
>
> 目标：一句话说清本轮要完成什么。
>
> 请读取以下上下文：
>
> - #文件A
> - #文件B
>
> 本轮约束：
>
> - 必须保留什么（例如集合主题、篇间关系、关键人物设定）
> - 不能做什么（例如不新增事实、不改篇序、不突破题材边界）
>
> 输出要求：
>
> - 写回到哪个文件或目录
> - 需要几个版本（如主版 + 短版）
> - 聊天侧只回摘要，不贴全文

### 6.2 最小输入清单（防止"喂不饱"）

每次至少给这四类信息中的三类：

1. **对象**：处理哪个短篇集、哪几篇文章。
2. **依据**：#全集总纲、#篇目卡、#正文、#人物、#审阅意见。
3. **目标**：本轮要提升什么（抓力、追读、篇间节奏、平台适配等）。
4. **约束**：哪些不能动（集合主题、篇序、核心冲突、字数门槛）。

### 6.3 工作目录与技术准备

> 这里的"工作目录"不是本项目根目录，而是你这个短篇集自己的目录，例如：`她不再回头-短篇集/`。

#### 6.3.1 创建短篇集工作目录

示例目录：`D:\小说项目\她不再回头-短篇集\`

#### 6.3.2 在短篇集工作目录下创建 junction（Windows）

进入 `她不再回头-短篇集/` 后，执行以下 4 条：

```powershell
mklink /j .github 本项目中对应题材下的.github目录路径
mklink /j 通用skills 本项目中的通用skills目录路径
mklink /j 写作研究 本项目中对应题材下的写作研究目录路径
mklink /j scripts 本项目中的scripts目录路径
```

例如当前题材是女频爱情，则 `.github` 与 `写作研究` 应指向 `女频爱情/` 下对应目录。

#### 6.3.3 推荐短篇集工作目录结构

创建后应包含以下子目录：

```
她不再回头-短篇集/
  小说总大纲.md           # 全集定位与承诺
  分辑大纲/                # 可选，用于分辑编排
    分辑一.md
    分辑二.md
  短篇清单/
    篇目卡.md              # 题目、冲突、回报、结尾策略
    优先级队列.md
  小说正文/
    01-篇名一.md
    02-篇名二.md
  人物传记/               # 跨篇人物档案（可选）
  技术设定/               # 跨篇规则或设定（可选）
  审阅意见/
    全集审阅/
    单篇审阅/
  竞对分析/
  投稿组包/
  .github/               # 指向本项目题材目录的.github
  通用skills/            # 指向本项目的通用skills
  写作研究/              # 指向本项目题材目录的写作研究
  scripts/               # 指向本项目的scripts
```

#### 6.3.4 在 VS Code 中打开短篇集工作目录

后续执行应使用：

- **斜杠命令**：如 `/竞对分析`、`/设计大纲`、`/创建小说正文`
- **上下文引用**：如 `#小说总大纲.md`、`#短篇清单/篇目卡.md`、`#小说正文/01-篇名一.md`

不是写 `女频爱情/.github/prompts/xxx.prompt.md` 这种路径命令。

### 7.5 架构段：/大纲审阅优化闭环（篇目卡回炉）

> /大纲审阅优化闭环
>

>

> 输出：
> - 单篇题目卡审阅写入 #审阅意见/篇目审阅.md
> - 调整后的 #短篇清单/篇目卡.md

### 7.6 编排段：/设计大纲（前置篇序与阅读动线）

> /设计大纲
>
> 目标：在正文开写前先完成全集篇序编排，明确阅读动线与节奏分布。
>
> 请读取：
>
> - #小说总大纲.md

> - 压轴篇（最后 1~2 篇）：高潮、升华或留白，完成整体收束
>
> 输出：
>
> - #分辑大纲/篇目编排方案.md
> - 给出"首篇、转折篇、压轴篇"选择理由
> - 标注"篇间冷热线"与"读者追读欲望曲线"预测

## 8. 短篇集高频误区与修正

1. **只顾单篇爽点，不顾全集统一性**
   - 修正：每 3~5 篇做一次全集层审阅，防风格漂移。

2. **选题池重复（同一种冲突反复写）**
   - 修正：题目卡必须标“冲突类型标签”，避免高重复。

3. **每篇都像长篇开头，缺短篇收束力**
   - 修正：提前定义结尾策略（反转/留白/回扣/悬停）。

4. **分发时一稿多投不分化**
   - 修正：固定“母稿 -> 平台适配 -> 输出”顺序，逐平台复核。

5. **篇间冷热线失控或连续性中断**
   - 修正：维护跨篇人物/线索台账，标记每篇开场回顾 + 结尾埋点，防止某篇掉档或读者遗忘。

## 9. 运营回流与迭代（短篇集版）

短篇集发行后，应该建立持续反馈闭环，为下一个集合或平台优化积累数据。

### 9.1 收集多平台读者反馈

逐平台汇总：

- **完读率**：各篇读者完成比例（反映篇长感知）
- **评论热度**：哪些篇章产生讨论，讨论核心是什么
- **转发与收藏**：哪些篇章传播力强（反映代表性）
- **读者评价**：高频词汇、情绪倾向、改进建议

### 9.2 识别强弱篇章与模式

找出：

- **强篇章**：完读率 > 80%、高评论、高转发 -> 分析成功因素
- **弱篇章**：完读率 < 60% -> 分析失败原因（冲突太弱？结尾不满意？与集合主题脱离？）
- **意外爆款**：计划之外获得高关注 -> 可作为下一集合的借鉴

### 9.3 回写并迭代下一个集合

基于数据：

- 下一集合的题材选择（读者期待什么？）
- 篇目设计（往强篇方向调整）
- 人物与设定（保留受欢迎的元素）
- 平台策略（哪个平台反馈最好，投入资源）

## 10. 最短执行清单（短篇集版）

1. `/竞对分析` + `/设计大纲`，完成立项与题目池（过 S1/S2）。
2. `/大纲审阅优化SOP`，把全集总纲与题目卡回炉到可执行（过 S3）。
3. `/创建小说正文` + `/审阅章节正文`，逐篇闭环生产（过 S4）。
4. `/设计大纲`（编排）完成篇序与动线。
5. `/评估起点签约概率` -> `/组包投稿材料` -> `/多平台输出编排`（过 S5）。

## 11. 快速迁移到其他题材

本指南框架可复用到其他短篇集题材（都市悬疑短篇集、异能志怪短篇集、女频爱情短篇集等）。迁移时，只需在以下几处做定制化修改：

### 11.1 题材约束与禁行项

| 约束维度 | 修改点 |
| --- | --- |
| **冲突类型** | Section 7.3 的篇目卡示例：改为该题材的常见冲突模板 |
| **禁行元素** | Section 6.2、Section 6.3 的技术准备描述：加入题材特定的审核红线 |
| **编排策略** | Section 7.6（前置篇序编排）与 Section 7.11（后置母稿提纯）需按题材做双阶段调整 |
| **风格边界** | Section 1 的 call architecture 说明：补充"本题材语言风格、写实度、暴力度"的标准 |

### 11.2 篇目卡设计的修改

- Section 7.3 示例：用该题材的 TopN 榜单作品进行竞对示例
- 评估标准：替换为该题材的"强冲突"特征定义
- Section 7.6 前置编排：按题材重排“首篇、转折篇、压轴篇”的分布逻辑

### 11.3 平台分发策略调整

- Section 7.11 的母稿提纯侧重点：按题材调整“信息密度/情绪强度/解释比例”
- Section 7.12 的平台选择：根据题材在各平台的热度排序
- Section 9 的反馈关键词：预先设定该题材在各平台的高频用户评论方向

### 11.4 审阅重点的微调

- Section 7.10 的"短篇紧凑度"标准：根据该题材的平均篇长调整
- Section 7.6 的"篇序与动线"标准：根据题材确认起势节奏与中段分布
- Section 8 的误区：补充该题材特有的常见坑（如悬疑集的逻辑漏洞、异能集的设定冲突）

## 12. 结语

短篇小说集的本质，不是“很多短篇”，而是“有辨识度的作品系统”。

只要坚持两点：

1. **入口始终用 `/命令`，上下文始终用 `#引用`。**
2. **单篇质量闭环 + 全集一致性闭环同时运行。**

你就能把短篇集从“素材堆”升级成可持续迭代、可分发、可投稿的完整产品。
