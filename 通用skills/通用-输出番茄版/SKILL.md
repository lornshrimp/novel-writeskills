## 番茄平台爆款写作实操要点（知乎高赞经验精华）

1. **书名要长且爆**：
	- 标题必须用一句话吸引人，题材、人物、身份、金手指、爽点、反差点等元素排列组合，标题即卖点。
	- 例：只看标题就能明白故事核心，拒绝抽象、无钩子的短标题。

2. **开头必须爆点**：
	- 前7行/300字务必抛出冲突、反差或危机，绝不允许慢热或大段设定。
	- 让读者一眼看到“异常/危机/反常/损失”，直接丢进矛盾现场。

3. **调动情绪，拉扯感强**：
	- 小说是放大生活，要用极端、放大的情节和冲突激发读者情绪，而非平铺直叙。
	- 每章/每段都要有情绪刺激点，不能只靠自嗨或平淡叙述。
	- 例：让主角遭遇极端不公、反转、打脸、爽点，制造“必须追下去”的情绪波动。

4. **关注数据，动态调整**：
	- 持续关注完读率、书架比、评论互动等数据，主动调整内容和节奏。
	- 研究榜单和热门题材，动态优化创作方向。
	- 与其纠结稿费，不如专注提升故事力和读者认可度。

5. **持续学习与成长**：
	- 先学后写，反复修正，关注数据，持续提升故事力和平台适配度。
	- 坚持不一定有回报，但不坚持一定没回报。

# 通用-输出番茄版

这是平台共性本体，负责承接“番茄版输出”的跨题材共性规则。

## 平台实战经验与成长建议（知乎高赞答主经验迁移）

### 1. 收入与现实门槛
- 番茄稿费与完读率、字数、推荐量强相关，单靠拼字数无效，需高完读率和持续追更。
- 大部分新人收入有限，爆款极少，需理性预期。

### 2. 平台机制与成长路径
- 首秀、推荐验证、书测、全勤奖等环节有明确门槛（如十万字完读率15%+、日更要求），流量池分级，断更/低完读率会被降权。
- 黄金三章决定生死，前三章完读率直接影响后续流量。

### 3. 读者画像与内容偏好
- 以三线及以下城市、低收入、低学历为主，偏好快餐文学、直给爽点、情绪拉扯、强钩子、短段落、快节奏。
- 文笔要求低但情节抓人，书名需长且直给，卖点、身份、金手指、反差等元素组合。

### 4. 写作与适配建议
- 首章前300字/7行必须有爆点或冲突，300-800字一抓点，章末强钩子。
- 多用短句、感官细节、情绪刺激，避免大段设定和自嗨。
- 关注数据反馈，及时调整内容和节奏，避免断更，持续提升故事力。
- 合理安排更新频率和字数，确保完读率门槛达标。

### 5. 常见误区与成长建议
- 杜绝只拼字数、气氛拖字、无大纲注水、频繁断更等低效写作方式。
- 先学后写，反复修正，关注数据，持续提升故事力和平台适配度。
- 坚持不一定有回报，但不坚持一定没回报。

## 默认执行顺序

1. 先锁定源稿不可变层：事实、事件链、证据链、角色关系与因果顺序不得改坏。
2. 读取对应题材的 `题材名-输出番茄版` Skill，确认题材入口与路由关系。
3. 再读取本 Skill 的 `references/平台共性执行细则.md` 与 `references/分节级补救映射与详细规则回填.md`，执行番茄平台的节奏强化、抓点布置、章末硬钩子与读屏优化。
4. 由题材包装层继续读取其题材 references，锁定题材边界、题材禁行项与题材专项补丁（作为兜底约束）。
5. 在“平台优先、题材兜底”前提下执行改写正文，确保平台化表达与源稿保真同时成立。
6. 最后才处理降相似度，不得为了降重破坏移动端抓力与章末追读驱动。
7. 执行 POV 契约复核，确保与 `platformPovContract` 一致，不得出现未授权人称漂移。
8. 改写输出完成后，必须立即执行“输出后自检与修订”节的五项自检，不得跳过。
9. 若自检未通过，必须按结论回炉修订并复检，直至“自检通过/自检修订完成”后方可落盘。
## POV 契约与连续性（强制）

- 若本次由 `通用-多平台输出编排` 调度，必须继承该流程已锁定的 `platformPovContract`，不得在本 Skill 内重新决定人称。
- 若用户直接命中本 Skill，则必须在改写前先从源稿正文与当前平台已完成前序章节中推断并锁定 `platformPovContract`；默认优先级为：用户明确指定 > 当前平台已完成前序章节 > 源稿既有 POV 链路 > 本批次首个已通过章节。
- 未获批准不得把连续章节从第三人称静默改成第一人称，或反向漂移；若源章本来就是视角切换章 / 多视角连续章，只能按已登记 `switchPlan` 执行，不得临场换壳。
- 落盘前后都必须显式执行 POV 校验，并把结果写入日志或执行记录；中文平台默认使用 `scripts/pov_validate.py` 或 `scripts/run_pov_gate.ps1` 以 `lang=zh` 或 `auto` 复核正文区块，第三人称链路检查对话外第一 / 第二人称，第一人称链路检查对话外第一人称锚点。

## POV 选择指南（如无显式契约）

若本次执行未获得上游明确的 `platformPovContract` 锁定，按以下指南为本平台适配稿选定**全书统一人称**：

> **人称是全书级契约**：所选人称必须贯穿本平台所有章节，不得逐章切换，不得章内混用（对话引用除外）。极特殊情况下某章须切换主视角人物时，该章开头必须明确声明「我是[角色名]」，且整章只保持该角色的第一人称叙事。

### 番茄版的 POV 定位

**平台特点**：免费阅读、强钩子、事件驱动、追更率优先

**默认推荐**：**第三人称（全书统一）**

番茄平台的核心驱动是密集的外部事件链。「有限全知 + 多线POV」叙事策略本质上是第三人称下的快速多角色场景切换；第三人称便于多人物动态的即时呈现和密集事件推进，与番茄高完读率、高追更率的目标高度吻合。

**适合选择第三人称（全书统一）的情形**：
- 事件链节奏快、多角色互动密集，需要快速展示多人物现场反应
- 免费平台强情节驱动，外部事件密度须远高于心理独白

**适合选择第一人称（主角第一，其他角色第三）的情形**：
- 源稿为第一人称，且个人视角是题材核心卖点
- 注意：番茄第一人称须严格控制心理独白比例，保持事件密度和外部动作驱动

## 默认输出口径

- 默认输出一版可直接继续落盘或进入平台终检的番茄派生正文。
- 默认保留原章核心事件链、快反馈节拍与章末硬钩子，不新增关键事实。

- 保住原章核心事件链、追读驱动与移动端可读性。
- 读起来不像泛免费平台稿，也不像把正文剪成摘要段子。

## 输出后自检与修订（必做，不得跳过）

改写输出完成后，必须立即执行本节自检；根据发现的问题就地修订，确认通过后方可落盘。**不得以任何理由跳过本步骤。**

**五项自检：**

1. **零新增事实**：改写版未引入源稿中不存在的关键事实、人物或设定；引入则删除/还原。
2. **核心事件链完整**：起因→推进→高潮→章末钩子方向均在改写版中体现；断裂则补回。
3. **平台风格达标**：改写版满足本 Skill“最低交付”（或平台核心风格要求）所列各项；不足则就地修订。
4. **无模板句污染**：未出现连续 2 句及以上解释腔/模板腔/口径名词句群；命中则整段回炉。
5. **POV 一致**：改写版 POV 与已锁定的 `platformPovContract` 一致；漂移则回炉修正。

**结论格式（必须输出）：**

- 全部达标：`[自检通过] 五项均达标，可落盘。`
- 已修订达标：`[自检修订完成] 已修订：<问题描述>，现五项均达标，可落盘。`
- 未通过（不得落盘）：`[自检未通过] 命中：<问题描述>，需继续处理。`

命中“未通过”后必须修订并重新输出自检结论，循环直至输出“通过”或“修订完成”方可落盘。

## 硬规则

- 绝对禁止输出“抽象自述 + 动词模板 + 口径名词”的垃圾句群；凡出现类似“对照项落在… / 先把同一句话拆碎 / 先把顺滑的解释拆开 / 只求能追溯（复核、对得上）”等模板化短语，必须判定为污染并整稿回炉。
- 严禁使用或间接调用以下脚本对正文做生成、扩写、拼接或降重：`scripts/append_cn_unique_monologue.ps1`、`scripts/append_cn_unique_narration.ps1`、`scripts/append_cn_unique_thirdperson.ps1`、`scripts/rephrase_cn_body.ps1`、`scripts/rephrase_en_body.ps1`、`scripts/cn_lexicon_profile_transform.ps1`。这些工具不得进入平台正文生产链路。
- 平台稿新增“模板句污染清零门禁”：正文不得出现连续 2 句及以上“我/我这边/我心里… + 把/将… + 对齐/校准/锁死/拆开… + 连接词 + 口径名词 + 只求/不求/先把…”结构；命中即失败，不论字数与相似度是否通过。
- 平台连续章必须服从已锁定的 `platformPovContract`；不得在本 Skill 内擅自把前序稳定链路改成人称新链路。
- 若命中未授权 POV 切换或 POV 校验失败，必须按 `pov_drift_detected` / `pov_switch_without_approval` 判定失败并整稿回炉。

- 命中本技能时，必须同时加载对应题材的 `题材名-输出番茄版` Skill。
- 题材特有规则不得回写到本文件中平行维护。
- 修改番茄平台共性规则时，应优先修改本 Skill，而不是多个题材入口。
- 不得把历史迁移参考里的 `都市悬疑` 口径直接当成跨题材默认规则继续扩散。
- 平台派生正文默认落在本 Skill 的工作目录 `番茄小说/` 下。
- 章节级派生正文目录统一按小说结构决定：若作品有分部，则位于 `番茄小说/第X部/第Y卷/`；若作品无分部，则位于 `番茄小说/第X卷/`。

- 章节派生正文必须以 `.md` 文件形式落盘：若对应章节目录不存在，执行前必须先创建完整目录路径，再将改写后的章节内容写入该目录下的 `.md` 文件，不得只输出聊天稿而不写入文件。
- 落盘路径的完整推断优先级：① 用户显式指定 chapterPath → ② 从源文件名与分部/卷信息自动推断 → ③ 若无法确定分部/卷，暂停并向用户确认，不得乱推断后静默落盘。
- 平台稿落盘后必须显式运行 `scripts/count-chapter.ps1 -MinCJK 3500 -MaxCJK 6500` 核正文，并运行 `scripts/count-afterword.ps1 -MinCJK 200 -MaxCJK 300` 核 `## 作者有话说`。
- 生成阶段必须先达到字数保底目标再允许落盘：正文目标 `4200–5800` CJK（硬下限 `3500`），`## 作者有话说` 目标 `220–280` CJK（硬下限 `200`）；未达保底目标时必须先回炉扩写，禁止“先落盘再补字数”。
- 在无损迁移完成前，不得删除源 `输出番茄版.prompt.md` 中尚未完成映射的内容。
