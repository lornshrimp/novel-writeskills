# 多平台输出执行总流程与断点恢复

## 输入归一化

支持以下输入：

- `chapterPath`
- `chapterPaths[]`
- 正文粘贴（仅在必要元数据可判断时）
- `platforms`（可选，默认全平台）

默认全平台集合：

- 番茄
- 七猫
- 知乎
- 豆瓣
- 出版社
- 微信订阅号
- 百度百家号
- 小红书
- B站
- 新浪微博
- 纵横小说
- 今日头条
- My Fiction
- GoodNovel
- WebNovel

## 路径与编号推断

### 章号优先从文件名编号前缀解析

- 标准形态：`{部.卷.章} 标题.md`
- `chapterNo`：完整编号前缀
- `chapterIndex`：编号最后一段

若无法解析 `chapterNo`，才追问。

### 分部 / 分卷目录推断

- `{分卷目录}` = 原文件所在目录名
- `{分部目录}` = `{分卷目录}` 上一级目录名

若路径无法稳定推断，才追问最小必要信息。

## 平台 POV 契约预检（新增硬性）

### 源稿 POV 形态先判定

- 在建立章节任务队列时，先判断每个源章属于：单视角稳定章 / 视角切换章 / 多视角连续章。
- 若源章本来就是视角切换章或多视角连续章，继续服从 `通用-创建小说正文` 的视角规则；多平台输出层不得擅自把它简化成另一套 POV，也不得把单视角章无理由改成人称迁移章。

### 平台契约的建立与继承顺序

- 每个平台都必须建立一份 `platformPovContract`，至少包含：`expected`、`lang`、`baselineSource`、`switchPolicy`。
- 优先级：用户明确指定 > 同平台已完成前序章节 > 源稿既有 POV 链路 > 本批次该平台首个已通过章节。
- 若目标平台目录下已存在同书同分部 / 分卷的前序章节，必须优先读取最近一个已通过章节；必要时补看最近 2–3 章，确认该平台链路是否稳定后再续写。
- 一旦平台契约锁定，后续连续章节默认全部继承；不得每章重新决定第一 / 第三人称。

### 允许切换的唯一场景

- 源章本来就是视角切换章或多视角连续章，且平台版保留这条结构；
- 用户明确要求该平台做 POV 迁移；
- 历史平台链路已经稳定使用另一套 POV，本次只是沿着既有链路续写。

除上述场景外，任何中途切成人称新链路都记为 `pov_drift_detected` 或 `pov_switch_without_approval`。

## 总执行顺序

1. 批量预检或单章预检
2. 建立章节任务队列
3. 读取原章并提炼主信息
4. 抽取源稿 POV 形态并为各平台建立 / 恢复 `platformPovContract`
5. 先做平台化推演，再分平台落稿
6. 每个平台写后先跑门禁、POV 校验与回炉
7. 全部目标平台稳定落盘后，再做分语言终检
8. 写入摘要并把日志标记为 `completed`

## AI 不可用时的强制降级策略（新增）

- 适用场景：子代理额度限制、模型不可用、调用持续失败。
- 唯一允许动作：
  1) 立即停止正文生成；
  2) 将日志状态写为 `blocked_ai_unavailable`；
  3) 记录失败调用点、失败原因、下一次恢复入口；
  4) 仅允许继续执行“读取、门禁、日志整理”等只读或检测动作。
- 明确禁止：切换到任何脚本改写链路去生成正文（包括批量替换、模板拼接、自动扩写、规则改写）。
- 命中该场景时，不得宣称“任务完成”，只能输出阻塞回执与恢复步骤。

## 先推演后落笔（硬性）

- 对每一个“章节 × 平台”节点，必须先形成平台化推演草图，再进入真正的正文改写与落盘。
- 推演草图至少应回答：
  - 这章主信息里哪些必须保留；
  - 该平台最优先的语体 / 节奏 / 钩子 / 标题倾向是什么；
  - 本次准备采用哪几种降重动作；
  - 哪些表达绝不能沿用中性底稿或源章原句硬套。
- 推演草图只用于执行，不进入最终文件，不在聊天中长篇展示。
- 若缺少这一步，就不允许直接进入平台正文写作；“先写中性稿，再批量轻改”视为流程违规。

## 允许中断的唯一场景（硬性）

- 无法从原文件名解析 `chapterNo`
- 无法稳定推断 `分部目录 / 分卷目录`
- 用户只粘贴正文且必要元数据缺失
- 真实执行错误导致下一步无法继续
- AI 生成能力不可用（子代理限额 / 模型不可用 / 调用失败）

除上述场景外，不得因为“汇报进度”“确认偏好”“下一步提醒”而暂停等待用户回复。

## 并行规则

### 默认保守上限

- 同时最多推进 **2 个章节任务**
- 每章内部默认最多 **2 个平台分支** 并行
- 全局活跃平台分支总数默认不超过 **4**

### 并行边界

- 5.1 原章读取完成前，不得启动任何平台分支
- 平台内必须串行：推演 → 标题 → 正文 → 落盘 → 门禁 → 回炉
- 任一平台未通过门禁，不得抢跑到终检
- 任一章节未完成全部目标平台，不得宣称该章完成

### 多章节滚动策略

- 多章任务必须先做一次性预检：路径存在性、编号解析、目录推断、平台集合归一化。
- 若多章中存在硬性缺参，必须一次性汇总缺失项后再询问，禁止逐章来回追问。
- 建立任务队列后，默认按输入顺序滚动推进；除非用户明确要求，不得自行重排章节顺序。
- 同一分卷的共享设定、平台门禁口径、日志模板与摘要模板可以复用，但每章都必须独立提炼主信息、独立终检。
- 某一章节失败，不得污染其他章节已通过的节点；失败隔离必须做到以章节为边界。

## 日志机制

### 日志目录

- `SOP执行日志/多平台输出/`

### 单章 logKey

- `chapterPath + platforms(归一化排序)`

### 多章与正文粘贴 logKey

- 多章：`normalized(chapterPaths，按输入顺序去重后拼接) + platforms(归一化排序)`
- 正文粘贴：`source=inline + chapterNo + 分部目录 + 分卷目录 + platforms(归一化排序)`

### 续写规则

- 查找最近一份 `status != completed` 且 `logKey` 相同的日志
- 若存在：续写并从断点恢复
- 若不存在：新建日志

### 日志最小字段

- `sop`
- `executionId`
- `logKey`
- `status`
- `startedAt / updatedAt`
- `inputs`
- `derived`
- `sourcePovPlan`
- `platformPovContracts`
- `checkpoint`
- `outputs`
- `similarityCheck`
- `platformValidate`
- `povCheck`
- `errors[]`

### 建议的批次与章节字段

- 顶层 `derived` 至少包含：`chapterCount`、`currentChapterNo`、`currentPlatform`
- 顶层 `checkpoint` 至少包含：`phase`、`nextAction`
- 顶层应能直接回答“当前平台 POV 契约从哪里来、是否允许切换”，至少保留 `sourcePovPlan` 与 `platformPovContracts`
- 多章模式建议使用 `chapters[]` 承载每章状态，每个章节条目至少包含：
  - `sourcePath|sourceLabel`
  - `derived.chapterNo / chapterIndex / 分部目录 / 分卷目录`
  - `sourcePovPlan / platformPovContract`
  - `checkpoint`
  - `status`
  - `outputs`
  - `platformValidate`
  - `povCheck`
  - `textQuality`
  - `similarityCheck`

### 已完成节点复用规则

- 续跑时，凡是“章节 × 平台”节点已经满足 `written=true` 且对应门禁字段完整、状态为通过，必须直接跳过。
- 续跑时不得覆盖已锁定的 `platformPovContracts`；若本次判断与旧契约冲突，必须先复核来源，再决定是否批准切换，而不是直接把契约改掉。
- 只有以下情况才允许重跑已写入节点：
  - 该章终检明确要求回炉；
  - 日志字段不完整，无法确认其是否真正通过；
  - 用户明确要求整体重写该节点。

### 并行写日志合并规则

- 若存在多个平台分支并行推进，任何一次写回日志前都必须先读取最新版本，再合并本分支负责的字段后整体覆盖写回。
- 禁止基于旧副本盲写，冲掉其他平台的 `outputs / gates / checkpoint / errors / nextAction`。

### 多章节规则

- 每个章节一份子日志
- 可额外维护一份批次主日志
- 禁止用一份混合日志承载多个章节的细粒度平台状态

## 章节完成与批次完成判定

- 单章：全部目标平台写入完成且通过门禁、POV 一致性检查、文本质量扫描、风格一致性检查与分语言相似度终检后，才能记为 `completed`。
- 多章：只有当每一章都独立完成上述终检后，批次顶层状态才能写为 `completed`。
- 若存在失败章节或阻塞章节，顶层状态必须保留 `failed` 或 `blocked_missing_input`，并把失败节点与下一步动作写入日志。
