# 多平台输出执行总流程与断点恢复

## 输入归一化

支持以下输入：

- `chapterPath`
- `chapterPaths[]`
- 正文粘贴（仅在必要元数据可判断时）
- `platforms`（可选，默认全平台）

默认全平台集合：

- 番茄
- 七猫
- 知乎
- 豆瓣
- 出版社
- 微信订阅号
- 百度百家号
- 小红书
- B站
- 新浪微博
- My Fiction
- GoodNovel
- WebNovel

## 路径与编号推断

### 章号优先从文件名编号前缀解析

- 标准形态：`{部.卷.章} 标题.md`
- `chapterNo`：完整编号前缀
- `chapterIndex`：编号最后一段

若无法解析 `chapterNo`，才追问。

### 分部 / 分卷目录推断

- `{分卷目录}` = 原文件所在目录名
- `{分部目录}` = `{分卷目录}` 上一级目录名

若路径无法稳定推断，才追问最小必要信息。

## 总执行顺序

1. 批量预检或单章预检
2. 建立章节任务队列
3. 读取原章并提炼主信息
4. 先做平台化推演，再分平台落稿
5. 每个平台写后先跑门禁与回炉
6. 全部目标平台稳定落盘后，再做分语言终检
7. 写入摘要并把日志标记为 `completed`

## 并行规则

### 默认保守上限

- 同时最多推进 **2 个章节任务**
- 每章内部默认最多 **2 个平台分支** 并行
- 全局活跃平台分支总数默认不超过 **4**

### 并行边界

- 5.1 原章读取完成前，不得启动任何平台分支
- 平台内必须串行：推演 → 标题 → 正文 → 落盘 → 门禁 → 回炉
- 任一平台未通过门禁，不得抢跑到终检
- 任一章节未完成全部目标平台，不得宣称该章完成

## 日志机制

### 日志目录

- `SOP执行日志/多平台输出/`

### 单章 logKey

- `chapterPath + platforms(归一化排序)`

### 续写规则

- 查找最近一份 `status != completed` 且 `logKey` 相同的日志
- 若存在：续写并从断点恢复
- 若不存在：新建日志

### 日志最小字段

- `sop`
- `executionId`
- `logKey`
- `status`
- `startedAt / updatedAt`
- `inputs`
- `derived`
- `checkpoint`
- `outputs`
- `similarityCheck`
- `platformValidate`
- `errors[]`

### 多章节规则

- 每个章节一份子日志
- 可额外维护一份批次主日志
- 禁止用一份混合日志承载多个章节的细粒度平台状态

## 允许中断的唯一场景

- `chapterNo` 无法解析
- `分部目录 / 分卷目录` 无法推断
- 用户只粘贴正文且必要元数据缺失
- 真实执行错误导致下一步无法继续

除上述场景外，不得中途暂停等待“继续”。
