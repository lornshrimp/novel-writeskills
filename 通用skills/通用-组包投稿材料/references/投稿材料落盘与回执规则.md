# 投稿材料落盘与回执规则

## 默认落盘原则

- 除非用户明确要求“只草拟不落文件”，否则必须把结果写入仓库。
- 优先维护可直接查看、可继续迭代的正式文件，不要把完整投稿包只停留在聊天窗口。

## 起点内投包默认路径

- `小说简介/起点内投/《作品名》_起点内投材料_vYYYY-MM-DD-rN.md`

## 单文件默认结构

单文件组包版默认按以下结构分节：

1. `## 组包口径与材料基础`（可选但强烈建议）
2. `## 当前包定位（正式投稿版 / 可投包预备版）`（可选但强烈建议）
3. `## 内投版简介`
4. `## 内投版正文（至少6000字）`
5. `## 内投版大纲（至少2000字）`
6. `## 组包自检`
7. `## 当前最强与最弱一环`（可选但强烈建议）

其中，以下 4 节仍视为固定核心交付：

1. `## 内投版简介`
2. `## 内投版正文（至少6000字）`
3. `## 内投版大纲（至少2000字）`
4. `## 组包自检`

落盘建议：

- `## 组包自检` 默认优先套用 `references/组包自检模板与风险矩阵模板.md`。
- 若对应题材 Skill 已提供题材化 `起点内投材料落盘模板.md`，则在通用结构基础上优先套用题材模板。

## 允许拆分的文件形态

若用户明确要求拆分文件，也可拆为：

- `小说简介/起点内投/《作品名》_起点内投简介_vYYYY-MM-DD-rN.md`
- `小说简介/起点内投/《作品名》_起点内投正文_vYYYY-MM-DD-rN.md`
- `小说简介/起点内投/《作品名》_起点内投大纲_vYYYY-MM-DD-rN.md`

## 默认版本策略

- 优先写入单文件组包版，便于一次查看和后续迭代。
- 若已有同名历史版本，默认追加新版本小节，不直接覆盖旧版本。
- 只有用户明确要求覆盖，或当前文件本来就是统一维护的主文件时，才改为覆盖式更新。
- 若本轮仍属“可投包预备版”，应在文件内显眼写明，不得让历史版本看起来像已经正式过线。

## 对话侧最小回执

完成后至少说明：

- 写入了哪个文件
- 本轮是覆盖还是追加
- 当前包是正式投稿版还是可投包预备版
- 本轮是否吸收了已有平台评估 / 审阅结论
- 当前包最需要继续补强的地方

## 完成底线

- 没有实际落盘，不算正式完成
- 没有说明落盘路径与版本策略，不算回执完整
- 没有指出最弱一环，不算完成收口
- 没有区分正式投稿版 / 可投包预备版，不算状态标识完整
