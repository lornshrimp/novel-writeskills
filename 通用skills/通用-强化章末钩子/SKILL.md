# 通用-强化章末钩子

章末钩子不是一句漂亮问号，而是把角色推到更糟的位置，让读者不翻下一章就难受。

## 保真迁移要求（强制）

命中“章末钩子强化 / 结尾翻页感强化”任务时，除本文件外，**必须继续读取**：
- `../通用-强化章节开头/references/钩子类型分类与样本库.md`（**必读** — 了解六大钩子类型与实战样本）
- `references/章末钩子强化规则.md`
- `references/章末钩子检查清单.md`
- `references/章末补丁回写控制卡与场景职责表.md`
- `references/腾讯专栏章末卡章方法增补.md`
- `references/腾讯专栏卡章引催更与段评互动增补.md`

## 常见触发词 / 用户说法速查

- 帮我救一下结尾
- 这章末没翻页感
- 最后一句不够尖
- 只修最后 150 到 300 字
- 下一章驱动太弱了
- 这结尾像总结不像钩子

## 何时使用

- 结尾像总结，不像钩子。
- 最后一段有情绪，但没有“下一章不得不看”的推动力。
- 章末太像收尾、抒情、平稳落地或结构总结。
- 用户只想强化章尾，不想重动整章。

## 不适用情形

- 本章真正问题在开头不抓或中段无回报。
- 你想靠新增事实硬造爆点。
- 你要做整章的钩子—回报—再钩子统筹，而不是只修结尾。

## 章末补丁前的裁判源复核（强制）

- 动手改最后 `150–300` 字前，必须认真仔细阅读与本章实际出场人物对应的人物传记，以及与本章背景、技术、规则、证据、程序链、世界观有关的技术设定文档。
- 不得只凭结尾片段、上一章印象或题材惯性硬抬钩子；凡涉及人物选择、关系翻面、设定代价、证据翻面、规则误判的章末落针，都必须回到作者侧裁判源核对。
- 若同一人物传记或技术设定存在多个版本，必须先确认当前可执行的最新版本；若关键裁判源缺失，必须先补齐或定位缺失的裁判源文件，或明确阻断，不能边猜边补最后一下。
- 这里的“补齐”只指补齐作者侧裁判源、定位缺失资料或确认阻断，不等于向正文新增事实。
- 章末强化后若改变了人物下一步动作、代价显影方式、规则解释方向或线索指向，必须回看一次相关裁判源，确认没有把下一章接棒点修歪。

## 题材竞对分析接入要求（强制）

- 若当前任务服务某个题材，且该题材目录下的 `竞对分析/` 已存在与当前项目、当前平台或当前赛道相关的竞对分析报告，强化章末前必须先读取并纳入结尾判断。
- 若目标平台为起点中文网，或未注明平台但默认按起点口径处理，且当前题材目录下尚无可用竞对主报告，且本章属于**卷尾章 / 首个收费节点 / 关键转折章 / 大回报章 / 重大翻面章**之一，强化章末前应优先调用 `通用-分析竞对作品`，至少补齐 **Top10 候选池、`3–5` 本主样本与四层深拆速览**；若任务限制导致本轮无法补齐，必须显眼记录盲区，并降低章末强化强度判断。
- 不得把这些报告只当成背景资料；章末补丁必须显式说明本次准备在哪些维度追平、反超竞对，以及哪些旧坑必须避开。
- 章末强化的默认目标不是“更尖更狠”而已，而是让当前作品至少在**续读驱动、新代价显影、信息翻面、讨论点、卖点接棒**这 `5` 个默认关键维度里形成可举证的追平路径，并尽量在其中 `2–3` 项建立明确领先方案。

## 起点化竞对吸收口径（新增）

当章末服务起点中文网，或未标平台而默认按起点基线处理时，默认把竞对吸收拆成以下四层，并翻译成最后 `150–300` 字补丁约束：

- **市场数据层**：用榜单位次、均订 / 首订 / 收订比代理信号、推荐资源位等，反推当前章末在续读驱动、收藏 / 追订意愿与下一章接棒强度上至少该达到什么强度。
- **内容创作层**：重点吸收主样本在新缺口、新代价、未完成动作、更危险新解释、关系翻面与新线索落针上的做法，把它翻成最后一段和最后一句的具体补丁。
- **运营策略层**：重点吸收主样本如何在章末制造本章说讨论点、催更点、段评引用点与卷尾传播点，把它翻成动作、口径差、物件、截图、来电、门响等可见落针。
- **受众反馈层**：重点吸收本章说 / 书友圈 / 书评区里高频被夸的卡章方式、被骂的虚空狠话、老白读者厌烦的作者预告腔，提前写进章末避坑与强化方案。

起点语境下，默认尽量使用以下术语，而不是写成空泛的“结尾更有翻页感”：`均订`、`高订`、`首订`、`追订`、`收订比`、`分类月票榜`、`畅销榜`、`本章说 / 段评`、`书友圈`、`老白读者`。拿不到精确值时必须写成代理信号，不得伪造具体数字。

## 本 Skill 的核心任务

1. 强化最后 **150–300 字**左右的翻页冲动。
2. 让结尾尽量落在以下之一：
   - 新缺口
   - 新代价
   - 未完成动作
   - 更危险的新解释
3. 避免把结尾写成空感慨、主题升华、作者旁白总结或解释句收尾。
4. 在驱动下一章的同时，继续给全书主卖点供血，而不是换成与主卖点脱节的噱头。
5. “下一章驱动”只允许作为作者侧判断标准，不允许在正文里直接说“下一章会……”“后面会……”“我们会看到……”。

## 执行顺序

固定优先级：先判结尾病灶与信息边界，再只修章末局部，最后才判断是否需要回写上游。

0. 若按起点口径执行且当前题材目录下缺少可用竞对主报告，先调用 `通用-分析竞对作品`，补齐 `Top10 候选池 → 3–5 本主样本 → 四层深拆` 的最小结论集。
1. 先圈定本章最后一段到最后约 150–300 字。
2. 若按起点口径执行，先吸收竞对主样本四层结论，并写清本次章末补丁要追平 / 反超的关键项。
3. 识别当前结尾问题属于哪类：
   - 太圆
   - 太虚
   - 太解释
   - 太早收情绪
   - 结尾只是总结
   - 有危险感但太空
   - 有事但不够尖
4. 保住原有结尾里的：最关键情绪点、最后一条信息推进、本章与下一章的连接口。
5. 优先做的动作：
   - 用未完成动作收尾
   - 用字段、截图、话术、来电、动作、门响、脚步、口径差、新变量落针
   - 让代价或危险更具体
   - 删掉解释性收束
6. 若修完结尾仍发现最后一场职责、中段回报或下章接棒对象发虚，先用“章末补丁回写控制卡与场景职责表”判断是否回写上游。
7. 只强化结尾，不顺手把全章重写；若确认是上游职责问题，再回写控制卡或最后两场职责，而不是继续硬抬最后一句。
8. 复检：读者是否会自然产生“下一章呢”的心理惯性，但不会被提前剧透答案。

## 默认输出口径

- 默认优先只修改最后 200 字；若钩子跨两段，才连同前一小段一起处理，但仍只限章末局部。
- 若用户指定“只改最后一句”，则缩小范围。
- 默认不新增正文事实，只对当前已在正文或裁判源中成立的信息重新落针。
- 默认直接回写结尾对应段落。

## 推荐公式

1. 本章已有结果刚刚落地。
2. 更糟后果紧跟着出现。
3. 留一个未完成动作或新变量。
4. 收在“下一步必须做”的压力上。

## 章末钩子分类（新增）

- **新危机型**：刚解决一层麻烦，立刻冒出更具体的新代价或新风险。
- **未完成动作型**：角色已经做出决定或伸手去做，但动作还没完成就截断。
- **新解释型**：旧信息被突然改写，读者必须追下一章确认解释是否成立。
- **关系翻面型**：信任、站队、亲密、误解、身份位差在结尾发生突然偏转。
- **新线索落针型**：最后一句落下一个足够具体的新变量、物件、口径差或时间窗。

默认优先从本章已经铺好的信息里选钩子类型，而不是临时硬造爆点。若本章承担单章闭环职责，章末最好形成“钩子 → 蓄力 → 爆发 → 留钩”的完整四拍，而不是只剩一句虚空狠话。

## 跨章节钩子链与回收节律补强（新增）

- 单章钩子不是孤立句子，默认要放回跨章节链里看：上章的问是否被回应、本章结尾又把什么问题 / 动作 / 代价交给下一章。
- 默认尽量做到“回应一个旧悬念，再打开一个新悬念”；若连续多章只开新坑不回旧坑，读者会产生被吊着走的疲劳。
- 悬念窗口默认参考：小悬念 `1–3` 章、中悬念 `5–8` 章、大悬念跨卷 / 全书；有意延迟响应最好不要超过 `3` 章。
- 同时活跃的高关注悬念线最好控制在 `3–4` 条；过多会稀释翻页动机。
- 钩子链响应模式默认区分：直接连续 / 间接关联 / 有意延迟 / 断裂；断裂通常视为结构事故。
- 最近 `3` 章尽量轮换钩子类型，不要连续多章都用“突然揭示”或“紧急危机”一招鲜。
- 若一章把所有事都交代完了，优先尝试把收尾截在“倒数第二个事件”或“未完成动作”处，把最后一下真正的结果留给下一章。
## 极其快的节奏与极其强烈的冲突——章末钩子强化方法（新增）

### 快节奏结尾的特征与规则

**1. "未完成动作"作为钩子**
快节奏的结尾应该让读者看到一个"正在进行中但被截断"的动作，而不是一个"已经完成"的状态：
```
❌ 慢：陈勉走进房间，看到了坐在对面的人。他的心里闪过一阵冰冷。他慢慢坐下。

✅ 快：陈勉推开门。对面的人已经站起来了，手里拿着一把刀。陈勉的手伸向腰间，但他意识到自己没有带枪。
```

快版本在结尾留下了一个"手伸向腰间但没枪"的未完成动作，读者必须追下一章看"接下来怎么办"。

**2. "倒计时"的后退式加速**
快节奏的结尾应该让读者感到"时间在流逝，下一秒就会发生某事"：
```
检查清单上还有 7 分钟。
收件箱里新来了一条信息：炸弹已启动。
警报声突然响起。
```

这些都是"时间在压迫、事情在加速"的信号。

**3. "新信息翻面"的突然改写**
快节奏的结尾可以在最后一刻改写读者对某件事的理解，强制推动下一章阅读：
```
陈勉以为这是一次谈判。直到他看到对面的人脸上的笑容。那不是谈判者的笑，那是猎人的笑。
```

"猎人的笑"这个新信息突然改写了整个局面。

### 强冲突结尾的特征与规则

**1. "代价的即时呈现"**
强冲突的结尾应该让读者清晰地看到：这个选择或失败的具体代价是什么：
```
❌ 虚：陈勉知道自己已经失败了。

✅ 实：她的身体从楼梯上摔下去。陈勉的手是最后一次接住她，但现在，她已经在五楼的地面上了。而陈勉还站在十楼的楼梯口。
```

强冲突的结尾必须具体到"谁失去了什么"。

**2. "权力的瞬间易位"**
强冲突的结尾应该让读者感到权力在这一刻发生了根本的翻转：
```
五分钟前，陈勉还可以选择。现在，选择的权力已经不在他手里了。警察已经包围了整栋楼。他能听到楼下的警犬在吠。
```

权力的易位能制造最强的冲击感。

**3. "三元冲突的升级"**
强冲突的结尾应该让读者看到：这个两方冲突突然变成了三方甚至多方的复杂局面：
```
陈勉以为只有他和绑匪在对抗。但警方的子弹打穿了绑匪的肩膀，绑匪倒下的一瞬间，陈勉才意识到——警方的狙击手一直都在这里。现在，陈勉既面对警方，也面对失控的绑匪。
```

突然的"第三方介入"能迅速升级冲突的复杂度。

**4. "代价的不可逆性"**
强冲突的结尾应该明确表现：这个后果已经无法改变了：
```
他按下了按钮。爆炸声响起。建筑在摇晃。这不是可以撤回的决定了。
```

"无法改变"这四个字隐含在每一个动作和后果里。

### 快节奏 + 强冲突的结尾模板

```
## 第 X 章末钩子强化

**当前结尾的功能分析**
- 这章承担什么职责（闭环 / 升级 / 回报 / 铺设）
- 下一章需要什么前置信息或时间状态
- 这章的冲突是否已经进入最后爆发阶段

**快节奏结尾的三要素**
- 未完成的动作（什么被截断了）
- 倒计时或时间压力（还有多少时间）
- 新信息翻面（什么理解被改写了）

**强冲突结尾的三要素**
- 具体代价（谁失去了什么）
- 权力易位（谁掌握了主动权）
- 冲突升级（从二元升到三元或多元）

**改写后的最后 300 字应该完成**
- [ ] 上一段的情绪或行动是否还在推进中（未完成感）
- [ ] 是否有明确的时间压力或倒计时感
- [ ] 代价是否具体且即时（不是"会有问题"，而是"已经失去了 X"）
- [ ] 权力分布是否发生了可感知的改变
- [ ] 是否避免了心理总结或作者旁白
```

### 执行时的快速检查清单（快冲突结尾专项）

[ ] 结尾是否以"未完成的动作"或"即将发生的事"截断，而非"已经完成的状态"？
[ ] 最后一段是否有明确的时间压力或倒计时感？
[ ] 代价是否具体而非抽象（"已经失去 X"而不是"可能有问题"）？
[ ] 权力分布是否在结尾发生了可感知的改变？
[ ] 是否有新信息在结尾突然翻转了局面或理解？
[ ] 冲突是否在升级（从简单对抗升到更复杂的多方局面）？
[ ] 是否避免了总结、预告、解释或旁白？
[ ] 结尾是否与本章前面的钩子形成了"钩子-蓄力-爆发-留钩"的四拍？
[ ] 是否有明确的"下一步必须做"的推动力，而不是虚悬？
[ ] 钩子是否与本书主卖点对齐，而不是随意的转向？
## 最低交付

- 结尾必须落到具体动作、具体后果或具体新变量。
- 读者应能感觉到：如果不看下一章，会错过一件马上要发生的大事。
- 若本章承担卖点回报职责，结尾还应把下一次卖点强化自然牵出来。
- 结尾的信息张力必须高于本章中段，而不是平着结束。

## 硬规则

执行时只按以下三组顺序检查：先信息与事实边界，再钩子表达边界，最后卖点与范围边界；前一组未过，不进入后一组。

- 先守信息与事实边界：
   - 不得靠新增关键正文事实硬造反转。
   - 未认真仔细阅读本章相关人物传记与技术设定 / 背景设定文档前，不得硬改结尾的人物选择、关系翻面、规则解释、证据指向与代价落点。
- 再守钩子表达边界：
   - 不得只靠问号或一句虚空狠话充当钩子。
   - 不得用“下一章会怎样”“后面会发生什么”“我们会看到”之类预告腔代替钩子；翻页感必须长在正文现有事件里。
   - 不得把结尾写成结构总结、主题总结或作者总结。
- 最后守卖点与范围边界：
   - 不得为了强钩子而破坏人物动机、口径与有限视角。
   - 不得把控制卡、后记或审阅报告里的“下章接棒点”原样翻译成正文最后一句。
   - 不得为了翻页感把结尾引到和全书主卖点无关的假热点上。

## 与其他 Skill / Prompt 的边界

- 本 Skill 专攻结尾钩子，不负责整章精修。
- 若用户需要整章的钩子—回报统筹，应交给正文润色类 skill。
- 若用户要强化开头，不由本 Skill 处理。
