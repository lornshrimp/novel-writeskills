# 通用-设计线索伏笔与回收台账

线索与伏笔不是装饰品，而是作者侧的追读承诺管理系统。

## 常见触发词 / 用户说法速查

- 帮我做线索台账
- 这批伏笔怎么记账
- 看看回收窗口有没有问题
- 这条误导是不是不可证伪
- 帮我整理伏笔和回收计划

## 何时使用

- 新建或重写线索台账、伏笔台账、回收总表。
- 审计现有台账是否存在“编号混乱、字段缺失、回收无代价”的问题。
- 需要把总纲、分卷、章节中的线索与回收动作对齐到统一账本。

## 默认执行顺序

1. 先判本轮对象属于线索台账、伏笔台账还是混合台账。
2. 再按字段模板建立编号、来源、状态、窗口、兑现后果。
3. 再补“提出 → 强化 → 回收 → 余波”的阶段链。
4. 最后检查能否直接回指到总纲、分卷与章级动作。

## 悬念管理与钩子链节律补强（新增）

- 线索台账不只记“埋了什么”，还要记“它属于哪类悬念 / 钩子链的哪一环、上次推进在哪一章、预期何时回应”。
- 默认尽量遵循“每章回应一个旧问题，再打开一个新问题”；若某条线连续多章只被提起不被推进，应标为失温或超期。
- 悬念窗口默认可分：小悬念 `1–3` 章、中悬念 `5–8` 章、大悬念跨卷 / 全书；超窗时要么补推进，要么写明合理延迟原因。
- 同时活跃的高关注悬念线最好控制在 `3–4` 条；超过时，台账里应明确主次与轮换顺序。
- 回收方式默认至少标记一种：对称场景、重复对白、物件回归、信息重组；若最终解答主要依赖末章新信息，默认判为不公平。
- 需警惕并显式标红：虚假悬念、机械降神、过度留白、低风险钩子、同类型钩子连续重复。

## 默认输出口径

- 默认输出一份可直接回指、可跨卷追踪的线索 / 伏笔台账。
- 默认同时补齐编号、状态、窗口与兑现后果。

## 最低交付

- 每条线索/伏笔都有唯一编号与当前状态。
- 每条回收都写清回收窗口、证据载体或触发动作。
- 每条兑现都写清“爽点 / 代价 / 余波”三项中的至少两项。

## 硬规则

- 禁止只有“这里埋个伏笔”，没有回收窗口。
- 禁止只有“后面回收”，没有回收动作与兑现后果。
- 禁止把误导写成无法证伪的作者遮羞布。
- 禁止同一编号跨文件漂移却没有同步口径。
- 百万字以上项目必须显式区分短线、中线、长线伏笔，并写明强化窗口。
- 终局级谜底不得主要依赖末章首次出现的新信息；关键信号应可回溯到前文。

## 与其他 Skill / Prompt 的边界

- 本 Skill 负责线索、伏笔与回收账本本体。
- 不直接代替总纲、分卷或案件引擎设计。
