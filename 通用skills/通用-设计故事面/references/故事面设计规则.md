# 故事面设计规则

## 一、故事面的最低定义

故事面不是题材标签，也不是概念摘要。

它必须同时是：

- 全书级冲突发动机
- 卷级供血面
- 章节抓手池
- 审稿裁判表

## 一点六、来源层速判（新增）

- `冲突发动层`：一句话谜面、六要素、硬抓手、节点责任与代价回声是否成立。
- `卷级供血层`：卷级推进、4A / 4B、供血锚点、主卖点持续兑现是否成立。
- `分卷可拆层`：故事面能否真正下压到分卷主问题、卷尾钉子、章群落点与正文抓手。

默认顺序：先立 `冲突发动层`，再判 `卷级供血层`，最后看 `分卷可拆层`；不要先谈供血，连冲突发动机都没立住。

## 一点五、与 `小说总大纲.md` 的边界（强制）

- 故事面默认主要承载支线故事；它不是 `小说总大纲.md` 的注释页、摘要页或拆分副本。
- 一个合格故事面必须能独立讲成一个完整故事，同时说明它怎样推动主线、又怎样被主线反推。
- 与主线共享人物、事件、线索、阶段节点通常属于正常联动；只有机械重复同一信息、且没有新增职责差异时，才算冗余。
- 若故事面只能复述主线节点，却说不出自己的冲突发动机、阶段胜负、代价与交接钉子，就说明这条支线还没有被设计完成。

## 二、固定骨架

1. 一句话谜面
2. 六要素
3. 护栏与验收
4. 卷级推进
5. 4A 出场卷与交接总览
6. 4B 卷内完整故事
7. 供血锚点
8. 写作检查清单

## 三、必要元素及其作用

### 1. 硬抓手

没有硬抓手，故事面就无法进入正文。

典型硬抓手包括：日志、哈希、编号、台账、用章、附件、工单、元数据、条码、版本号、登录痕、封存袋、在场固定。

### 2. 节点责任

不要骂抽象体制，而是追“谁签、谁盖、谁导出、谁放行、谁退回”。

### 3. 代价回声

每卷都写代价，防止主角零成本推进、反派零成本作恶。

### 4. 交接硬点

故事面不是本卷结清，而是把下一卷要追的那颗钉子交出去。

### 5. 可写化结构

主卷详写，副卷简写，避免总纲很强、正文失速。

## 四、经验教训

- 概念必须立即压回载体。
- 失败要证据化。
- 不要让对手无限制开挂。
- 主卷要写完整战役，副卷只留任务。
- 余痛类素材必须少而狠。

## 五、验收口径

一个故事面通过验收，至少要满足：

- 能看出这一面控制哪种胜负
- 有明确的 A/B/C/D 抓手层
- 有 ≤3章破口
- 有主出场卷 / 阶段与副出场卷 / 阶段
- 有至少 3 条跨卷硬抓手线
- 能明确说明它与 `小说总大纲.md` 的交接接口，而不是只做主线摘要
- 有至少 1 个主卷可直接拆章
- 有检查清单可反复使用

## 五点五、故事面最小工件（新增）

```markdown
【冲突发动层】一句话谜面 / 六要素 / 硬抓手 / 代价回声
【卷级供血层】卷级推进 / 4A4B / 供血锚点 / 卖点兑现
【分卷可拆层】分卷主问题 / 主抓手 / 卷尾钉子 / 章群落点
【当前判定】Green / Yellow / Red
```

## 六、Green / Yellow / Red 故事面判定（新增）

- `Green`：三层都成立，故事面既能独立讲成故事，也能持续给分卷和正文供血。
- `Yellow`：概念与供血已有，但可拆性或卷级承重仍偏弱，需要继续压实。
- `Red`：只有议题 / 设定 / 情绪，没有真正的冲突发动、供血机制或分卷接口。
