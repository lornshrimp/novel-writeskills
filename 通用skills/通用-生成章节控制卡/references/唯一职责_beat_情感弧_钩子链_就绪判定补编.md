# 唯一职责_beat_情感弧_钩子链_就绪判定补编

> 吸收外部 `planner-skill`、`beat-vocabulary` 与 `plan-template` 的章节规划方法，把控制卡从“能看懂”升级为“能直接交给正文执行”的施工文档。

## 一、本章唯一职责

控制卡开头必须先写一句：

`本章的职责是让读者______。`

### 通过标准

- 一句话能说清
- 落点可被读者直接感知
- 不包含两个互不从属的独立任务

### 常见好例

- 让读者开始怀疑某角色的真实身份
- 让读者体验主角的第一次失败
- 让读者意识到上一章的胜利其实是误判

### 常见坏例

- 介绍背景并推进主线并深化人物
- 既要解释设定又要铺关系又要给高潮

一句话说不清，默认说明本章承担太多职责，应拆章或聚焦；直接判 `Red` 倾向。

## 二、不应包含

每张控制卡都要单列：

- 本章不展开哪条线
- 哪些解释属于后续章节
- 哪些情感或设定不能在本章提前摊牌
- 哪些段位只保留触发，不完成结论

这不是可选项，而是防止章节越权、抢跑、抢戏的硬边界。

## 三、节奏权重与疲劳检查

### 权重字段

- `重`：信息密度高、冲突激烈、情绪强、回报重
- `中`：常规推进、关系推进、人物深化
- `轻`：缓冲、着陆、转承，但也要有有效触碰

### 硬规则

- 禁止连续 `3` 章及以上都为 `重`，除非写明调整理由
- 轻章不是水章；必须承担整理、落地、情绪收束或下一轮蓄力职责

## 四、情感弧字段

每张控制卡必须写清：

- **读者情感入口**：进入本章时读者带着什么情绪
- **读者情感出口**：离开本章时希望读者处于什么情绪
- **情感弧模式**：从以下至少选一个
  - 紧张→释放
  - 困惑→清晰
  - 舒适→颠覆→新平衡
  - 渐进→爆发

若情感入口和出口几乎无变化，说明本章读感波形可能发平。

## 五、场景序列最小字段

每个场景至少写清：

- POV
- 场景目标
- 情感转变（从 X 到 Y）
- 参与角色
- 情感基调
- 冲突 / 张力来源
- 信息流
  - 读者获得了什么
  - 角色获得了什么
- 与下一场景的关系：因果 / 对比 / 并行 / 递进
- 过渡方式：时间跳跃 / 空间切换 / 情绪桥接 / 硬切
- 建议字数

## 六、Beat 规划规则

### 基本规则

- 每个场景由 `1–4` 个 beat 组成
- `承重 beat` 不可删除或随意改序
- `灵活 beat` 可合并、弱化或换实现方式
- beat 写“做什么”，不是“怎么写”

### 常用 beat 类型

- 开场类：`Hook / Scene-setter / Callback Opener / In Medias Res`
- 行动类：`Confrontation / Revelation / Discovery / Decision Point`
- 紧张类：`Complication / Counterattack / Steel-man Opposition / Tension Hold`
- 情感类：`Breather / Intensifier / Reflection Pause / Empathy Moment`
- 解决类：`Synthesis / Resolution / Aha Moment / Callback`
- 过渡类：`Bridge / Cliffhanger / Landing`

### 常见组合

- 开篇章：`Hook → Scene-setter → Confrontation → Cliffhanger`
- 推进章：`Callback Opener → Discovery → Complication → Decision Point → Cliffhanger`
- 高潮章：`In Medias Res → Confrontation → Revelation → Resolution`
- 揭秘章：`Hook → Discovery → Synthesis → Aha Moment → Callback`

## 七、悬念与伏笔规划

控制卡必须单列：

- 回应旧悬念：本章准备回哪条旧问题，怎么回
- 新增悬念：本章准备新开哪条问题
- 埋设伏笔：本章埋什么，以什么形式埋
- 章末钩子：类型、方向、与最近几章的差异

## 八、钩子链验证

每张控制卡都应补一段钩子链表：

| 项 | 内容 |
| --- | --- |
| 上章出口钩 | 上章钩子类型与方向 |
| 本章入口响应 | 直接延续 / 间接关联 / 有意延迟 / 漏回应 |
| 本章出口钩 | 本章钩子类型与方向 |
| 下章预期响应 | 预期下一章如何回应 |

### 处理规则

- 若上章钩子未回应且无合理理由，降级处理
- 有意延迟可接受，但最好在 `3` 章内回到这条线
- 最近 `3` 章尽量不要重复同一种钩子类型

## 九、就绪判定

### Green

- 唯一职责明确
- 场景职责清晰
- 承重 beat 完整且顺序合理
- 情感弧入口 / 出口已定义
- 不应包含写明
- 钩子链无断裂
- 字数分配与节奏权重自洽

### Yellow

- 缺 `1–2` 项非核心要素
- writer 可谨慎推进，但必须知道缺口在哪

### Red

- 唯一职责说不清
- 钩子链断裂且无说明
- 承重 beat 漏失
- 情感弧缺位
- 连续重章疲劳无调整方案
- 本章越权、把其他章节职责抢写了

### 自动 Red 红牌

- 连续 `3` 章以上 `重` 节奏且无调整理由
- 本章唯一职责无法一句话定义
- 钩子链断裂且无解释

## 十、交给 writer 的附注

控制卡尾部建议单列 `writer 注意事项`：

- 前章必须呼应的细节
- 人物当前状态的异常点
- 承重 beat 绝不能被润平的位置
- 哪些信息只暗示、不说破
- 哪些关系只推位差、不做最终结论

## 红线

- 不得只有“这一章大概会发生什么”而没有职责、beat 与就绪判定
- 不得只做章清单式摘要，跳过场景级字段
- 不得把钩子设计成“到时候看情况”
- 不得把 Red 章伪装成可直接开写的成熟控制卡
