# 通用-创建小说正文

章节正文创作是一个组合任务，不是把梗概摊平。

这是跨题材的正文创作母入口：当目标是**直接产出给读者阅读的章节正文**，并且要同时处理写前校验、正文落稿、章末钩子、`## 作者有话说`、`## 章节后记` 与文件回写时，应优先从这里进入。

## 黄金三章与开篇不铺垫（强制）

新手最常犯的开篇错误：大段文字讲解世界观、铺垫、人物介绍，这样直接劝退读者。

正确的开篇三章结构：
- **第一章**：抛出核心冲突。直接把主角扔进麻烦里，让读者跟着紧张。示例：「脚夫在城楼下，亲眼看着军阀屠了整个村子，他攥着扁担，第一次动了杀人的念头。」
- **第二章**：强化主角困境。形势更危急，压力更大。示例：「逃出生天后被追兵盯上，身边唯一的亲人重伤垂危，必须选择。」
- **第三章**：埋下核心悬念。一个足以牵扯出整个故事的秘密。示例：「死人堆里捡到刻着神秘纹路的腰牌，牵扯出乱世的惊天秘密。」

**千万不要铺垫** —— 直接把主角扔进冲突里。

## 核心驱动力设定（强制）

在开始落正文前，必须先问自己这个问题：
- **读者为什么要跟着这个人物走？他们在等什么？在担心什么？**

这就是"核心驱动力"——不是主角要去做什么，而是"读者跟着的理由"。它是一个**具体的问题**，而不是抽象的悬念。

示例对比：
- ❌ 错误：主角发现自己是失忆的（只是信息）
- ✅ 正确：主角发现自己是失忆的，而现在有人要杀他，他必须在24小时内想起自己是谁或者逃出去（这是问题，驱动读者往下看）

故事框架：**某人 → 渴望 → 遇阻 → 改变**
- 某人：一个具体的角色
- 渴望：他想要什么（要具体）
- 遇阻：什么力量阻止他（要有真实代价）
- 改变：他最终怎样改变（要不可逆）

## 何时使用

- 从分卷大纲 / 章纲 / 人物档案 / 设定文档出发，直接新写一章正文。
- 需要把已有草稿重写成可发布的完整章节。
- 需要在同一文件里稳定交付“正文 + 作者有话说 + 章节后记”。
- 用户明确要求直接写入目标章节文件，而不是只在聊天里给思路。

## 不适用情形

- 只是写前控制卡，还没到落正文阶段：优先使用 `通用-生成章节控制卡`。
- 只是检查连续性、冷热线和旧债回归压力：优先使用 `通用-管理连续性冷热线`。
- 只是修某一场戏或拆场景：优先使用 `通用-执行场景单元`。
- 只是抢救开头或结尾：优先使用 `通用-强化章节开头` 或 `通用-强化章末钩子`。
- 已有完整章节，只想精修：优先使用 `通用-正文润色`。
- 主要问题是模板腔或 AI 味：优先使用 `通用-去AI味重写`。

## 默认组合调用顺序

- 阶段 A：写前校验
	0. 若当前任务按起点口径执行，且本章处于**第一章 / 黄金三章 / 卷首章 / 卷尾章 / 首个收费节点 / 关键转折章**之一，或当前题材目录下已存在可用 `竞对分析/` 主报告，先读取并吸收 `Top10 候选池 → 3–5 本主样本 → 四层深拆` 的关键结论；若当前题材目录下尚无可用竞对主报告，优先调用 `通用-分析竞对作品` 补齐最小结论集，再进入正文施工。
	1. 先认真仔细阅读本章实际出场人物对应的人物传记，以及与本章背景、技术、规则、证据、程序链、世界观有关的技术设定 / 背景设定 / 控制卡，确认人物状态、能力边界、关系位置与规则代价。
	2. 先做资料校验与连续性扫描：`通用-管理连续性冷热线`
	3. 再把本章任务压成控制卡：`通用-生成章节控制卡`

- 阶段 B：正文施工
	4. 再把控制卡落成场景与正文：`通用-执行场景单元`
	5. 再用“场景串章与章级回报分配卡”确认哪一场负责抓眼、哪一场负责中段回报、哪一场负责章末钩子。
	6. 正文首轮必须把本章应有的场景、回报、升级与章末钩子一次写进正文，并把正文字数直接推到达标区间附近；只有已经明确记录结构性缺口时，才允许后续补写，不得把“后续再慢慢扩写”当常规流程。

- 阶段 C：定向补强
	7. 章首不够抓时，再调用：`通用-强化章节开头`
	8. 章末驱动力不足时，再调用：`通用-强化章末钩子`
	9. 全章需要整体提纯时，再调用：`通用-正文润色`
	10. 若 `## 作者有话说` 初稿偏复盘、讲设定、讲创作过程或剧透过重，再调用：`通用-润色作者有话说`
	11. 模板腔、假人声或机械表达明显时，再调用：`通用-去AI味重写`，并先做一句病灶诊断，再锁定三档手术强度。

- 阶段 D：最小复检清单
	12. 落稿后必须完成一组最小字数复检：正文运行 `scripts/count-chapter.ps1 -MinCJK 3500 -MaxCJK 6500`；`## 作者有话说` 运行 `scripts/count-afterword.ps1 -MinCJK 200 -MaxCJK 300`。
	13. 正文字数门禁只认 `count-chapter.ps1` 输出中的 `BodyCJK`、`Meets3500` 与 `WithinRange`；`Len` 仅作参考。
	14. `去空白统计`、`NoWhitespaceLen`、手工目测或任何自定义字符数，都不得替代脚本门禁字段。
	15. 若首轮落稿明显低于门槛，默认优先回到控制卡与场景职责补足“动作链 / 回报位 / 升级后果 / 章末钩子”，而不是用解释性段落做机械补字。

## 极其快的节奏与极其强烈的冲突——正文层实施方法（新增）

快节奏与强冲突不能只停留在大纲设计里，必须落实到每个场景、每段对话、每一句中。以下是可直接套用的正文执行规则。

### 快节奏的正文层实施

**1. "前 300 字"的多层信息密度**
每章的前 300 字应该用最少的词汇传达最多的信息：
- **是谁**：主角或关键角色出场（不用详细外貌，用一个最扎眼的识别符）
- **在哪**：具体的地点（不是"办公室"，而是"13 楼财务部的那间隔间，窗帘拉到一半"）
- **发生了什么**：眼下的冲突或异常（不是铺垫，而是"已经进行中的问题"）
- **有多危险/紧迫**：读者为什么要紧张（倒计时、威胁信号、身体反应）

示例（快节奏）：
```
陈勉推开财务部隔间的门时，王主任的脸色已经白透了。屏幕上的转账记录还在滚动——200 万，按每笔 10 万分出去，只用了三分钟。

"不止这个。"王主任颤抖地指向另一个窗口，"从年初开始。两年。"

陈勉的心沉了一下。两年。这不是临时挪用，这是有预谋的。

楼下，执法部队的车刚停在门口。
```

这 200 字里已经传达了：人物、地点、冲突（侵吞公款）、时间压力（执法部队要来了）、危险强度（从两年前开始）。

**2. 场景切换的"快速断推进"**
不要用过渡段描写角色的移动、思考或休整；直接切到下一个有冲突的场景：
- ❌ 慢：陈勉冲出办公室，冲进电梯，脑子里混成一团。这一刻他忽然明白了什么，转身要往楼下跑……
- ✅ 快：陈勉冲进电梯。楼下，执法部队已经登上了大厅。

快速切场能直接制造"不给主角（和读者）喘息时间"的压迫感。

**3. 对话里的"叠加信息"而非"轮流问答"**
每一轮对话都应该同时承载多个层面的信息，而不是纯粹的信息交换：
- 表层：谁说了什么
- 中层：说话时的态度转变或隐藏目的
- 深层：这句话对下一步行动的直接推动

示例（强对话）：
```
"两年。"王主任的声音里混着哭腔，"我想举报，但你知道的，他在上面有人。"

陈勉不说话。

"我没办法。"王主任继续说，声音更低了，"但如果有人敢动他……我可以作证。"

这不是示弱，这是威胁——王主任在告诉陈勉，如果官方动作太大，他会反咬。

陈勉抓起电话："我需要十分钟。"
```

每句话都在改变局面：从"我是受害者"→ "我也有隐形的权力"→ "我随时可以反水"。

**4. "动作-反应-后果"的紧凑三段式**
不要让一个动作的后果延迟显现；应该在同一场景内快速循环：
```
陈勉砸了手机。[动作]

王主任跳起来，往门外看。[反应 1]

对讲机里传来"已锁定财务部"的声音。[反应 2 - 并行的环境反应]

陈勉的脸色变了——他刚砸的手机里有定位器。[后果 1 - 代价显现]
```

这样的紧凑节奏能让读者感受到"每个动作都在改变局面，每个选择都有代价"。

### 强冲突的正文层实施

**1. 每场场景的"三元对立"设计**
一个有力的场景应该让至少三方的利益或立场发生碰撞，而不是简单的"主角 vs 敌人"：

示例：
- 主角想逃脱（目标 A）
- 搭档想自保，劝主角放弃（目标 B - 与主角冲突）
- 敌人在追，但也想抓活的（目标 C - 与主角冲突，但与搭档的目标有交集）

这三方的目标互相缠绕，导致"没有绝对的盟友，也没有绝对的敌人"的复杂局面。

**2. 对话中的"立场碰撞"而非"信息传递"**
任何对话都不应该是纯粹的Q&A；应该是两个诉求的碰撞：
- 甲的诉求：隐瞒真相，继续冒充
- 乙的诉求：揭露真相，得到补偿

对话的每一句都应该在争夺主动权：
```
"你为什么要背叛我？"陈勉问。

"背叛？"搭档冷笑，"我救了你的命。那笔钱本来是给刺客的，我拦了下来。"

陈勉愣了一下。

"但现在你欠我的。"搭档继续说，"要么跟我一起消失，要么我把你的身份卖给官方。三秒钟选择。"
```

这里每一句对白都在改变主权关系：从"被背叛者"→ "被救者"→ "被勒索者"。

**3. "人物目标与主角目标的冲突"的具体化**
不要让冲突停留在"立场不同"的抽象层面；应该具体到"这一刻，双方都想要同一个有限资源"：

**资源例子**：
- 同一条逃脱路线
- 唯一能证明清白的证据
- 掌握关键信息的那个人
- 有限的时间窗口

这样冲突就不是观点差异，而是**实实在在的竞争**：
```
证人只愿意见一个人，而且要在十分钟内。

陈勉和搭档都需要这个证人的证词，但理由完全相反：陈勉要证明自己无罪，搭档要栽赃陈勉。

两人同时冲向证人的住处。
```

这是**真正的冲突**——不是能商量的，也不是能各让一步的。

**4. "代价与选择"的不可逆性**
每个场景的结尾，主角都应该因为这场冲突而付出**不能撤销的代价**：
- 失去了一个信任的人
- 暴露了一个隐藏的筹码
- 激怒了一个之前不是敌人的势力
- 被迫做出违背原则的选择

示例：
```
陈勉逃了，但代价是让搭档被抓。

现在有两种可能：
1. 搭档会在刑讯中吐出陈勉的所有秘密
2. 搭档会用陈勉的秘密去和官方协商，牺牲陈勉来保护自己

两种都意味着，陈勉失去了唯一知道他全部真相的人。下一卷，他会被迫找新的信任对象，而新对象未必可信。
```

**5. 场景中的"权力位移"与"信息翻面"**
强冲突的场景应该让读者感受到**局面在剧烈改变**，而不是"说了很多话但局面没变"：

标记方式：
```
场景开始时：主角掌握主动权（知道敌人的位置，但敌人不知道他在哪）
场景中段：信息翻面（敌人突然出现，说"我们早就知道你在这里了"）
场景结尾：权力彻底反转（主角被包围，敌人要求投降或死亡）
```

每一次权力位移都应该以**具体的新信息**为触发点，而不是"敌人的运气更好"这样的虚弱解释。

### 场景字数与节奏的控制

为了确保"快"的体感，应该控制场景的字数与密度：
- **快速冲突场景**：1500-2500 字 / 场，信息密度高，动作快，对话有张力
- **复杂谈判 / 对话场景**：2000-3500 字 / 场，允许更多对话轮换，但每轮都必须改变局面
- **环境与逃脱场景**：1500-2000 字 / 场，短场景能制造"时间紧迫"的感受

禁止出现 4000+ 字的单场景，除非这个场景本身就是"卷级高潮"。长场景容易让节奏显得拖沓。

### 正文撰写时的"快冲突"检查清单

落稿时自检以下项目：

[ ] 前 300 字是否同时传达了人物、地点、冲突、紧迫感？
[ ] 场景切换是否避免了过渡段和休整时间？
[ ] 每轮对话都在改变局面（权力、信息、关系）吗？
[ ] 每个场景是否都有"动作 → 反应 → 后果"的紧凑三段式？
[ ] 是否有"三元对立"的多方角色冲突，而非简单的"主vs敌"？
[ ] 冲突的资源是否具体（路线、证据、时间窗口），而非抽象（立场、信念）？
[ ] 每个场景的结尾是否都有不可逆的代价或局面改变？
[ ] 是否有明确的"权力位移"或"信息翻面"时刻？
[ ] 场景字数是否合理（避免 4000+ 字的拖沓）？
[ ] 是否避免了"说了很多话但局面没变"的虚弱场景？

## 写前裁判源复核（强制）

- 正文落稿前，必须认真仔细阅读与本章实际出场人物对应的人物传记，以及与本章背景、技术、规则、证据、程序链、世界观有关的技术设定文档。
- 不得只凭章纲标题、上一章印象、题材套路或局部摘要直接开写；凡涉及人物动机、能力边界、关系状态、设定代价、证据机制的内容，都必须回到作者侧裁判源核对。
- 若同一人物传记或技术设定存在多个版本，必须先确认当前可执行的最新版本；若关键裁判源缺失，必须先补齐、定位或明确阻断，不能边猜边写。
- 若本章涉及多人、多条设定线或复合背景，必须把“谁的传记必须重读、哪份技术设定必须重读”列成最小清单，再进入正文施工。
- 正文、`## 章节后记` 或回写摘要里，至少要能说清本章具体服从了哪些人物传记约束、哪些技术设定约束，避免把裁判源读成摆设。

## 默认输出口径

- 默认直接产出可给读者阅读的章节正文成品。
- 默认同文件交付：正文、`## 作者有话说` 与 `## 章节后记`。
- 默认直接写回目标章节文件，而不是只在聊天里给草稿。

## 落盘目录层级（强制）

- 本 Skill 的默认工作目录固定为 `小说正文/`。
- 本 Skill 只处理章节级正文文件的创建、重写与回写，不承担分部 / 分卷独立文件的落盘规则。
- 若作品有分部，章节文件统一落在 `小说正文/第X部/第Y卷/`。
- 若作品无分部，章节文件统一落在 `小说正文/第X卷/`。
- 章节文件名默认不带日期，沿用项目既有章节号与短标题规则。

## 最低交付

- 有完整章节正文，而不是梗概或施工建议。
- 有清晰的章首抓力、中段回报与章末驱动。
- 有可直接保存的 `## 作者有话说` 与 `## 章节后记`，且 `## 作者有话说` 已写成“一个小细节 + 一点点共鸣 + 一个小问题”的读者向小剧场。
- `## 章节后记` 已按模板完整填写，而不是只写几句作者感想或施工备注。
- 首轮成稿默认应尽量直接过正文字数门禁，不把结构性缺口留给后续反复扩写。
- 能说明本章主卖点怎样被看见、强化或延续。

## 开篇、排版与单章节拍共性要求（新增）

- 前 `300` 字默认至少完成两件事：人物当前处境、眼前损失风险、一个必须马上处理的异常或冲突。
- 若当前属于第一章到第三章，默认还要检查黄金三章职责：第一章立人物抓手与第一麻烦，第二章让人物顶上第一个挑战，第三章必须给一次可感知的小高潮、反击或更强升级。
- 单章默认应尽量看出“**钩子 → 蓄力 → 爆发 / 回报 → 留钩**”四拍；哪怕某一拍偏轻，也不能长期缺失。
- 正文优先用动作、感官、物件、口气、停顿、错位反应来传情绪，不要用抽象总结和作者说理替代现场压力。
- 默认按手机端友好处理：句子能短则短，段落能轻则轻，连续大黑块要警惕，尤其是开头与回报位。

## 研究回灌补强（2026）

- 情感向长篇默认参照“黄金三章”闭环：第一章立危机与处境、第二章加压并给破局抓手、第三章给首次可感知回报或反击。
- 开篇前 `300` 字未见冲突 / 异常 / 压力时，默认视为起手风险；开篇前 `1000` 字至少应能看见一次明确回报预期。
- 单章优先维持“钩子 → 蓄力 → 爆发 / 回报 → 留钩”的读者节拍，避免只有说明没有动作、只有铺垫没有兑现。
- 正文表达优先“情绪具象化”而非抽象总结：通过动作、感官、空间反常、物件变化和对话错位来交付情绪。
- 情感相关章节默认可参考“爽点 / 苏点约 `3:1`”的密度控制，防止全章只有关系拉扯而缺乏推进回报。

说明：`## 章节后记` 为强制交付项，不适用“若适用”口径；执行时以 `references/章节附属构件与后记规范.md` 与下文“硬规则”为准。

## 题材竞对分析接入要求（强制）

- 若当前任务服务某个题材，且该题材目录下的 `竞对分析/` 已存在与当前项目、当前平台或当前赛道相关的竞对分析报告，落稿前必须先读取并纳入本章判断。
- 若目标平台为起点中文网，或未注明平台但默认按起点口径处理，且当前题材目录下尚无可用竞对主报告，写关键章节前应优先调用 `通用-分析竞对作品`，至少补齐 **Top10 候选池、`3–5` 本主样本与四层深拆速览**；若任务限制导致本轮无法补齐，必须显眼记录盲区，并降低本章放行强度。
- 不得把这些报告只当成背景资料；必须显式对比本章的抓力、回报、人物付费价值、信息差、钩子与读者记忆点，说明本章准备在哪些维度追平、反超竞对。
- 正文创作的默认目标不是“把这一章写顺”，而是让当前作品在关键维度上**全方位超越竞争对手**；若本章看不出明确超越路径，应继续返工而不是直接放行。

## 起点化竞对吸收口径（新增）

当正文服务起点中文网，或未标平台而默认按起点基线处理时，默认把竞对吸收拆成以下四层，并直接翻译成本章的落稿约束：

- **市场数据层**：用榜单位次、均订 / 首订 / 收订比代理信号、推荐资源位等，倒推本章在前 `300 / 1000 / 3000 / 6000` 字、章末续读驱动与题材归类效率上至少该达到什么强度。
- **内容创作层**：重点吸收主样本在开篇切入、场景冲突、人物抓手、规则边界、信息差、代价显影、章末钩子上的做法，把它翻成控制卡、场景职责与正文动作链，而不是只记一句“这本书节奏快”。
- **运营策略层**：重点吸收主样本的追读工程、回报密度、本章说讨论点、更新节奏匹配与卷首 / 卷尾功能位，把它翻成“章首抓眼—中段回报—章末留钩”的执行要求。
- **受众反馈层**：重点吸收本章说 / 书友圈 / 书评区里高频被夸的爽点、被骂的拖点、老白读者不耐烦的病灶，提前写进本章避坑与加码策略。

起点语境下，默认尽量使用以下术语，而不是写成空泛的“读者喜欢这个章”：`均订`、`高订`、`首订`、`追订`、`收订比`、`分类月票榜`、`畅销榜`、`本章说 / 段评`、`书友圈`、`老白读者`。拿不到精确值时必须写成代理信号，不得伪造具体数字。

## 本层职责

- 统一承接章节正文创作的通用输入、输出、流程和文件回写要求。
- 统一承接“正文 / 作者有话说 / 章节后记”的通用结构规范。
- 统一承接写前资源清单、输入模板、交付标准、自检与后记模板。
- 把开头、章末、整章精修、去 AI 味等专项问题路由到更专门的通用 Skill。
- 当正文要进入去 AI 味回炉时，默认把“需保留的钉子、推荐强度、主要病灶”一并交给 `通用-去AI味重写`，避免只喊一句“去味”。

## 与其他 Skill / Prompt 的边界

- 如果当前任务发生在某个题材目录，并且该题材已建立 `题材名-创建小说正文` 包装层，则应优先通过题材 Skill 进入，再由题材 Skill 路由到本通用 Skill。
- 本 Skill 负责跨题材稳定复用的正文创作骨架。
- 题材 Skill 负责题材读者口径、题材硬约束、题材技法、题材禁行项与题材样例。

## 硬规则

- 默认目标是**直接写出读者可读的章节正文**，而不是停在提纲、分析或施工建议上。
- 默认不采用“先写一个明显偏短的草稿，后面再靠多轮扩写补到门槛”的常规路径；首轮应尽量一次写到足够字数，只把补字留给小范围微调或结构性回炉。
- 除非用户明确要求“只做构思 / 不落文件”，否则应把结果直接写回目标章节文件。
- 开写前必须认真仔细阅读本章相关人物传记与技术设定 / 背景设定文档；若关键裁判源未读、未定位或版本未确认，不得凭常识补完后直接落稿。
- 正文必须服从作者侧裁判源（大纲、人物、设定、时间线、台账）；如有冲突，以裁判源为准。
- 正文落稿前，必须先确认本章如何呈现、强化或延续主卖点；不得把卖点写到大纲里、落稿时又丢掉。
- 章节前段与关键回报位必须尽快让编辑和读者感知到“这本书最值得继续看的地方”，不得把卖点埋到过晚位置。
- `## 作者有话说` 只允许做三件事：用一个贴着正文的小细节交换情绪、给读者一点现实共鸣、留下一个小问题；不得写成章节点评、创作总结、设定讲义或直白剧透。
- `## 作者有话说` 不得直接明示本章 / 本系列的核心思想、暗线结论或制度判断；包括“所有制”“阶级”“分配权”这类主题词，只可借问题与生活小场景引导读者自行联想。
- `## 章节后记` 必须至少覆盖：本章定位、功能点落地核对、冲突与悬念、爽点登记、伏笔与回收、设定/程序呈现、人物发展、证据要素、钩子类型、数字纪律检查、偏离记录、后续铺垫、读者吸引点与优化建议。
- `## 章节后记` 只做作者侧复盘与校验，不得替正文补剧情、补动机、补关键证据或补本该在正文里看见的回报。
- `正文` 属于读者沉浸区，严禁出现章节自指 / 连载导览 / 作者讲评式话术，如“这几章”“前几章”及其“前面几章 / 前面这几章”变体、“后几章”及其“后面几章 / 后面这几章”变体、“整章”及其“这整章 / 这一整章”变体、“上一章”“下一章”“本章 / 这章主要”“后文会”“我们会看到”“读者会发现”等；若要交代承接，只能改写为人物当下记忆、局势余波、身体后果、物件残留或未完成动作。
- 上述禁例按“模式族”理解，不以列举为穷尽；例如 `前/后(+面) + 这? + 几章`、`这?(一)?整章`、`(上/下)一章`、`(本/这)章` 都属于同类出戏表达。
- 章末驱动必须靠新变量、新后果、未完成动作或具体危险来成立，不得用“下一章……”之类预告句替代钩子。
- 不得把人物写成只有高光、没有短板；重要角色至少要让读者看见他在能力、情绪、关系或选择上的真实阻力。
- 章节正文文件名默认不带日期；若新建文件，仍必须服从上文的 `小说正文/` 工作目录与章节层级规则；`## 作者有话说` 与 `## 章节后记` 作为同一正文文件的组成部分，也不得额外改成日期版文件名，除非用户明确要求另存归档副本。
- 交付前不得只凭目测判断篇幅是否达标；必须显式运行 `scripts/count-chapter.ps1 -MinCJK 3500 -MaxCJK 6500` 校验正文，并运行 `scripts/count-afterword.ps1 -MinCJK 200 -MaxCJK 300` 校验 `## 作者有话说`。
- 若需要在后记、日志、摘要或审阅报告中记录篇幅数据，必须直接引用脚本输出字段 `Len / CJK / BodyCJK / Meets3500 / WithinRange / MeetsAll`；不得改写成“正文字数：XXXX（去空白统计）”之类的替代口径。
- Prompt 可以变薄，但执行时不得因为 Prompt 变薄而漏读本 Skill 及其 references。
- 若项目或题材层另有更严格的时间、编号、平台、视角或格式约束，必须继续服从题材层。

## 完成检查

- 是否已读取本 Skill 声明的 references
- 是否已认真仔细阅读本章相关人物传记与技术设定 / 背景设定文档，并确认使用的是当前可执行的最新版本
- 是否已按默认顺序组合调用上游 / 下游相关 Skill
- 是否已直接产出“正文 + 作者有话说 + 章节后记”完整结构
- 是否已确认本章卖点没有在落稿中失踪，并且在前段或关键回报位被有效看见
- 是否已确认 `## 作者有话说` 不是书评 / 复盘 / 设定讲义，而是“一丢丢创作花絮 + 一点点共鸣 + 一个小问题”的读者向小剧场
- 是否已确认 `## 章节后记` 的强制字段完整、与正文逐项对应，且没有拿后记偷补正文缺失内容
- 是否已对正文区块完成“章节自指 / 连载导览腔”复检，并清掉“这几章 / 前几章 / 前面几章 / 前面这几章 / 后几章 / 后面几章 / 后面这几章 / 整章 / 这整章 / 这一整章 / 上一章 / 下一章 / 本章 / 这章 / 后文会 / 我们会看到 / 读者会发现”等出戏措辞
- 是否已显式运行 `scripts/count-chapter.ps1 -MinCJK 3500 -MaxCJK 6500` 与 `scripts/count-afterword.ps1 -MinCJK 200 -MaxCJK 300`，并确认正文 / 作者有话说字数都符合门禁
- 是否已确认篇幅门禁判断使用的是 `BodyCJK`，而不是“去空白统计”或其它自定义字符数
- 是否已把结果写回目标文件，或明确遵从用户“不落文件”的要求
- 是否已完成自检，而不是只生成正文就停下
