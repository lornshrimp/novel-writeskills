# 通用-撰写书评

书评不是剧情复述，也不是创作教程，而是把“为什么值得点开继续读”写成读者愿意转发、平台愿意展示的非剧透安利文本。

## 继续读取的 references

- `references/平台与角度适配矩阵.md`
- `references/书评落盘与交付约束.md`
- `references/普通读者推荐型书评框架与种草切口.md`
- `references/按平台拆分的推荐型书评模板组.md`

## 何时使用

- 需要撰写全书、分部、分卷或分章书评。
- 已有书评太像剧情简介、太像模板评语，或已经剧透过度，需要重写。
- 需要针对不同平台或传播场景，生成不同口径的书评版本。
- 需要审计一篇书评是否真正站在读者视角、是否有钩子、是否能支撑传播。

## 书评命名规则（强制）

- 所有书评文件名都必须带日期，日期格式固定为 `YYYY-MM-DD`。
- 全书、分部、分卷书评统一放在 `书评/` 下，文件名采用“范围/标题 + `_书评_` + 日期”或既有约定的等价格式；例如：`书评/全书书评_2026-04-15.md`、`书评/第1部·日常裂缝_书评_2026-04-15.md`、`书评/1.1 这份 PIP 是死亡通知书_书评_2026-04-15.md`。
- 分章书评按所属分部、分卷存放在 `书评/第X部/第Y卷/` 下，文件名采用“章节号与标题 + `_书评_` + 日期”；例如：`书评/第1部/第1卷/1.1.1 HR 让我先签字再说_书评_2026-04-15.md`。
- 日期应位于文件名末尾、扩展名 `.md` 之前；除非用户明确要求只在聊天里草拟，否则不得省略日期版文件名。

## 默认执行顺序

1. 先确认书评层级、对应范围、目标平台与写作角度。
2. 再抽取可公开的卖点、阅读承诺、情绪曲线与适读人群，禁止复述完整因果链。
3. 再按平台语境和角度重点调整标题、段落气质、专业度与互动口径。
4. 再补写一则与正文呼应、但不重复剧透的 `# 短评`。
5. 最后按独立文件命名与落盘规则保存，并核查是否仍然站在普通读者视角。

## 最低交付

- 1 篇主书评正文。
- 1 则约 200 字的 `# 短评`。
- 明确的标题、平台口径与写作角度。
- 若涉及文件写回，必须给出符合规范的落盘路径与命名版本。

## 强制要求

- 命中本技能时，如当前任务来自题材目录，必须同时加载并使用对应的 `题材名-撰写书评` Skill。
- 禁止剧透关键反转、关键证据最终指向、角色生死结局或完整因果链。
- 禁止使用上帝视角、作者视角、讲授视角；必须保持“正在读 / 刚读完的普通读者”立场。
- 禁止把书评写成剧情复述、世界观说明书或空泛赞美词堆叠。
- 除非用户明确说明“只在聊天里草拟”，否则默认应按书评文件规则独立落盘，而不是只在对话里给一段文案。
- 一旦落盘，文件名必须遵守本 Skill 上述日期命名规则，不得生成无日期版书评文件。
- 若用户未特别指定，默认优先使用“真实体感 + 具体细节 + 阅读后劲”的普通读者推荐型口径。
- 若目标平台已明确，默认应套用对应平台的推荐型模板骨架，而不是只换几个词。
