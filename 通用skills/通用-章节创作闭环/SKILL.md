# 通用-章节创作闭环

章节创作闭环不是“写完正文就算完工”，而是一个带日志、回炉、评分门槛和读者产物的完整执行流程。

## 继续读取的 references

命中本 Skill 时，除本文件外，**必须继续读取**：

- `references/章节创作闭环总流程与完成出口.md`
- `references/章节路径推导与工件命名.md`
- `references/章节批量模式与连续性日志.md`
- `references/章节终审门槛与摘要规范.md`
- `references/单元主线编排与节奏配比.md`
- `references/九部件与进度节奏续跑卡.md`
- `references/章节链路闭环复核与回写决策卡.md`

## 何时使用

- 用户要求按 SOP 真实跑完整章或连续多章。
- 需要从大纲到终稿、再到审阅、读者笔记与分章书评的全链闭环。
- 需要支持断点恢复、批量模式、逐章门槛与最终摘要。

## 不适用情形

- 只写一章正文，不跑后续审阅与读者产物：优先使用 `通用-创建小说正文`。
- 只做审阅报告，不改稿：优先使用 `通用-审阅章节正文`。
- 只做某一步专项强化：优先使用对应专项 Skill。

## 常见触发词 / 用户说法速查

- 按 SOP 把这章跑完
- 从大纲一路做到终稿
- 这几章都按闭环来做
- 一直回炉到 9.2+
- 批量跑这几章
- 带日志续跑下去别停

## 默认组合调用顺序

1. `通用-创建小说正文`
2. `通用-正文润色`
3. `通用-去AI味重写`
4. `通用-审阅章节正文`
5. `通用-强化章节开头` / `通用-强化章末钩子`（按需）
6. 读者侧产物生成

## 默认输出口径

- 默认按完整 SOP 跑到摘要收口，而不是停在正文或单轮审阅。
- 默认同时落盘正文、终审报告、读者产物与日志状态。
- 若进入 `通用-去AI味重写`，默认要记录本轮采用的手术强度与主要病灶，避免回炉记录只写“已去味”。

## 章级节拍护栏

- 默认每章至少跑出一轮：**钩子 → 冲突 → 中段回报 → 新风险**，不能只做到“有事发生”。
- 连续 3–5 章最好形成一个可感知的小单元闭环，哪怕只是阶段性小胜、小识破或小翻盘。
- 若故事存在事实链 / 程序链 / 资源链等双轨推进，日志必须分别记录本章各自推进了什么，避免只剩单线热闹。
- “问题解决”不能直接抹平压力；本章解决一个具体问题后，默认还要带出一个更难的新问题、新代价或新窗口。

## 题材竞对分析接入要求（强制）

- 若当前任务服务某个题材，且该题材目录下的 `竞对分析/` 已存在与当前项目、当前平台或当前赛道相关的竞对分析报告，闭环开始前必须先读取并纳入本轮目标。
- 不得把这些报告只当成背景资料；必须在日志、终审结论或回炉判断中显式写出当前章节 / 当前批次相对竞对的领先点、持平点、落后点与补丁方向。
- 章节闭环的默认目标不是“文件齐了、分数过线”，而是推动当前作品在关键维度上**全方位超越竞争对手**；若看不出明确超越路径，不得把闭环误判为完成。

## 最低交付

- 有最新正文、同步更新的 `## 章节后记`、最新终审报告、读者笔记、分章书评与任务摘要。
- 有日志续跑状态、评分记录与回炉判断。
- 能说明本章或本批次是否真正闭环、是否仍需回写上游层级。

## 本层职责

- 统一承接章节 SOP 的步骤定义、默认路径推导、批量模式、日志机制、完成出口与回炉门槛。
- 统一承接“存在文件不等于步骤已执行”的校验口径。
- 统一承接逐章摘要与批量摘要模板。
- 统一承接步骤 0 初始化、工件命名、未分类兜底与连续性台账默认位置。
- 当章节创作出现卡顿、续跑失焦或返工过大时，先回到九部件与进度节奏层重建顺序，再继续正文链路，而不是盲目硬写。
- 在步骤 5–9.5 之间，统一判断这章是“真实闭环”还是“文件已齐但链路未闭”，并明确需要回写的上游层级。
- 若第 3 步去 AI 味只做了轻修但问题实为人物假、对话太顺或破局太机械，应明确标记需要升级到更高强度再回炉，不能把“做过去味”误判成“问题已解”。

## 工件命名规则（强制）

- 章节正文主文件默认不带日期；`## 作者有话说` 与 `## 章节后记` 仍附着在同一章节文件内，不单独创建带日期文件名。
- 分章终审报告与分章书评文件名必须带日期，日期格式固定为 `YYYY-MM-DD`。
- 多平台派生正文文件名默认不带日期，沿用平台目录与章节路径命名；日期只用于对应审阅报告与书评，不用于平台正文本体。

## 硬规则

- 只有当正文、终审报告、阅读笔记、分章书评、日志状态全部闭环时，才允许宣称完成。
- 任一章节正文在步骤 2 之后发生改动，都必须重跑字数与受影响审阅链：正文用 `scripts/count-chapter.ps1 -MinCJK 3500 -MaxCJK 6500`；若同文件含 `## 作者有话说`，再用 `scripts/count-afterword.ps1 -MinCJK 200 -MaxCJK 300` 复检该小节；不得只凭旧报告或目测放行。
- 任一章节正文在步骤 2 之后发生改动，都必须同步刷新同文件 `## 章节后记`，至少复核：本章定位、功能点落地核对、爽点登记、伏笔与回收、设定/程序呈现、人物发展、证据要素、钩子类型、数字纪律检查、偏离记录、后续铺垫、读者吸引点与优化建议。
- 步骤 3 / 4 / 5 / 6 / 7 / 8 / 9 不得因为“已有文件”或“分数看起来不错”被跳过。
- `9.2+` 的唯一判定，以**连续两轮独立终审报告中的综合评分均严格大于 9.2** 为准；首次过线后仍须追加一轮回炉修订与独立复审，第二轮复审也过线才允许收口。
- 命中本 Skill 时，必须坚持执行器身份，除硬性缺参外，不得把阶段汇报伪装成完成汇报。
- 每章日志必须记录主卖点保真状态，至少回答“本章怎样呈现 / 强化 / 维持卖点；若未显性呈现，下一章由谁接棒”。
- 每章日志还必须至少记录：章首抓手、中段回报、章末接棒点；若存在双轨推进，还要分别记录每条推进线新增了什么证据 / 动作 / 损失。
- 回炉判断不得只看文句顺滑度；若卖点在正文阶段变淡、变虚、变泛，也必须判为未闭环。

## 与其他 Skill / Prompt 的边界

- 本 Skill 负责编排与执行完整章节 SOP。
- 单步正文创作、审阅或专项强化，仍应交给对应的下游通用 Skill 本体处理。

## 完成检查

- 是否已读取全部必读 references
- 是否已对每章维护日志与一致性状态
- 是否已在日志或终审结论中记录本章卖点落地度与遗忘风险
- 是否已在本轮最终稿上重跑 `scripts/count-chapter.ps1 -MinCJK 3500 -MaxCJK 6500`，并在有 `## 作者有话说` 时同步重跑 `scripts/count-afterword.ps1 -MinCJK 200 -MaxCJK 300`
- 是否已确认同文件 `## 章节后记` 已按最新模板同步更新，而不是沿用旧版字段或旧版结论
- 是否已完成步骤 10 的读者产物与步骤 11 的分章书评
- 是否已按模板输出步骤 12 摘要
