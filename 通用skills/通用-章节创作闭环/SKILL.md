# 通用-章节创作闭环

章节创作闭环不是“写完正文就算完工”，而是一个带日志、回炉、评分门槛和读者产物的完整执行流程。

## 流氓启动法（强制心态）

**最核心的秘诀：先完成，再完美。破除三个常见幻觉：**

1. **"等灵感爆发"** — 真相是：小说是边写边长的，不是想出来的。
2. **"等大纲完整"** — 行动 → 反馈 → 迭代，而非"准备 → 准备 → 放弃"。
3. **"等文笔修炼到位"** — 90% 写不出小说的人，不是因为文笔不够，而是因为他们在等一个完美时机。

**策略就一句话：先开枪，再瞄准。** 

- 别等完美大纲、别追求开篇即封神、别死磕构思。
- 写着写着人物会活、情节会跑偏，这是正常现象。
- 与其花大半年构思和写大纲，不如先把初稿落地。
- **烂初稿也比空想强万倍**，而且是这些「不完美的初稿」推着你一步一步把想讲的故事落地生根。
- **现在就动笔，不要等。**

## 最小核心启动（强制）

- 若本轮写作仍处于“有意向但思路发散”状态，必须先产出一句“最小核心”：`谁在什么处境下，为了什么目标，正被什么阻力压迫`。
- 首章进入闭环前，必须明确“问题引擎”：读者看完第一章后，心里具体在追问什么。
- 闭环执行固定拆成 3 个独立动作：①先落稿；②再看反馈；③最后按反馈迭代。
- “继续准备”不是独立步骤；只要当前回合还没有实际落稿，就一律视为仍停留在第①步。
- 允许首轮粗糙，但不允许停在空想；日志里应记录本轮最小核心是否仍有效、问题引擎是否增强或偏移。

## 何时使用

- 用户要求按 SOP 真实跑完整章或连续多章。
- 需要从大纲到终稿、再到审阅、读者笔记与分章书评的全链闭环。
- 需要支持断点恢复、批量模式、逐章门槛与最终摘要。

## 不适用情形

- 只写一章正文，不跑后续审阅与读者产物：优先使用 `通用-创建小说正文`。
- 只做审阅报告，不改稿：优先使用 `通用-审阅章节正文`。
- 只做某一步专项强化：优先使用对应专项 Skill。

## 常见触发词 / 用户说法速查

- 按 SOP 把这章跑完
- 从大纲一路做到终稿
- 这几章都按闭环来做
- 一直回炉到 9.2+
- 批量跑这几章
- 带日志续跑下去别停

## 默认组合调用顺序

1. 先认真仔细阅读本章实际出场人物对应的人物传记，以及与本章背景、技术、规则、证据、程序链、世界观有关的技术设定 / 背景设定 / 控制卡 / 连续性台账，确认本章可写范围、不可越界红线与本轮必须兑现的约束。
2. `通用-管理连续性冷热线`
3. `通用-生成章节控制卡`
4. `通用-创建小说正文`（默认按“骨架稿 → 分场景扩写 → 拼合检查”执行）
5. `通用-正文润色`
6. `通用-去AI味重写`
7. `通用-审阅章节正文`
8. `通用-强化章节开头` / `通用-强化章末钩子`（按需）
9. 写后回到 `通用-管理连续性冷热线` 更新动态状态、悬念 / 伏笔状态与回归压力；若发生关键结构偏移，再补决策记录
10. 读者侧产物生成

## 规划—写作—审查—归档闭环补强（新增）

- 上游规划不是可选前戏：`通用-管理连续性冷热线` 与 `通用-生成章节控制卡` 默认属于闭环启动条件；若本章唯一职责、承重 beat、情感弧或钩子链仍发虚，不宜直接进入正文。
- 控制卡若已显性判 `Red`，默认先回炉再写；`Yellow` 可以谨慎推进，但必须把缺口写进日志，不能假装成熟度已经达标。
- 审阅环节默认不仅给分，还要区分 `writer / polisher` 处理方，并标问题来源层：`[骨架层] / [扩展层] / [表达层]`；否则闭环会把结构问题误丢给句子层修补。
- 真正的闭环不止“正文 + 审阅 + 去味”三件套，还包含定稿后的连续性 / 归档更新：人物动态状态、章节摘要、悬念 / 伏笔状态、必要的决策记录与最小进度快照。
- 每轮闭环日志最好至少能回答：当前处理章节、当前步骤、是否阻塞、最近完成章节、待处理问题、下步 `1–3` 个动作。
- 连续性与归档默认只记已发生事实，不预测未来章节；若定稿章节与旧资料冲突，以定稿章节为准更新，并在日志里写明修正原因。

## 闭环启动前的裁判源复核（强制）

- 闭环开始前，以及任一章进入润色、去 AI 味、审阅回炉前，必须认真仔细阅读与该章实际出场人物对应的人物传记，以及与该章背景、技术、规则、证据、程序链、世界观有关的技术设定文档。
- 不得只看章节标题、章纲摘要、控制卡摘录、上一章印象或题材惯性就开写、开改、开审；凡涉及人物动机、能力边界、关系状态、设定代价、证据机制的判断，都必须回到作者侧裁判源核对。
- 若同一人物传记或技术设定存在多个版本，必须先确认当前可执行的最新版本；若关键裁判源缺失，必须先补齐、定位或明确阻断，不能靠常识补完后继续闭环。
- 每章日志、终审结论或回炉记录里，至少要写清“本轮核对了哪些人物传记 / 技术设定、哪些约束已兑现、哪些仍待补丁”，避免闭环日志只有抽象进度。
- 任何环节一旦改动触碰人物动机、能力边界、关系状态、规则代价或背景事实，必须先做裁判源复核，再继续后续步骤。

## 默认输出口径

- 默认按完整 SOP 跑到摘要收口，而不是停在正文或单轮审阅。
- 默认同时落盘正文、终审报告、读者产物与日志状态。
- 若进入 `通用-去AI味重写`，默认要记录本轮采用的手术强度与主要病灶，避免回炉记录只写“已去味”。

## 章级节拍护栏

- 默认每章至少跑出一轮：**钩子 → 冲突 → 中段回报 → 新的未解决风险**，不能只做到“有事发生”。这里的“新的未解决风险”是指：本章结尾必须留下一个会抬高主角代价、并明确推动下一章接棒的新增压力点。
- 连续 3–5 章最好形成一个可感知的小单元闭环，哪怕只是阶段性小胜、小识破或小翻盘。
- 若故事存在事实链 / 程序链 / 资源链等双轨推进，日志必须分别记录本章各自推进了什么，避免只剩单线热闹。
- “问题解决”不能直接抹平压力；本章解决一个具体问题后，默认还要带出一个更难的新问题、新代价或新窗口。

## 研究回灌补强（2026）

- 闭环执行默认加一条快审链：`30秒（首屏）→ 3分钟（前三章）→ 首卷/前3万字`，任一层断裂都要回炉而非带病推进。
- 若连续多章推进，默认每 `3-5` 章应至少形成一次阶段性小高潮或关键兑现，避免“章章有事、章章不结”。
- 中期疲软预警：若连续若干章只做信息搬运、情绪平铺或重复冲突，必须插入“情绪加油站”（反差、短回报、关系新位）再续跑。
- 闭环评分除文句顺滑外，必须同时核验：开篇抓力、人物代价、章中回报、章末驱动、手机端可读性。

## 题材竞对分析接入要求（强制）

- 若当前任务服务某个题材，且该题材目录下的 `竞对分析/` 已存在与当前项目、当前平台或当前赛道相关的竞对分析报告，闭环开始前必须先读取并纳入本轮目标。
- 不得把这些报告只当成背景资料；必须在日志、终审结论或回炉判断中显式写出当前章节 / 当前批次相对竞对的领先点、持平点、落后点与补丁方向。
- 章节闭环的默认目标不是“文件齐了、分数过线”，而是推动当前作品在关键维度上**全方位超越竞争对手**；若看不出明确超越路径，不得把闭环误判为完成。

## 最低交付

- 有最新正文、同步更新的 `## 章节后记`、最新终审报告、读者笔记、分章书评与任务摘要。
- 有日志续跑状态、评分记录与回炉判断。
- 能说明本章或本批次是否真正闭环、是否仍需回写上游层级。

## 本层职责

- 统一承接章节 SOP 的步骤定义、默认路径推导、批量模式、日志机制、完成出口与回炉门槛。
- 统一承接“存在文件不等于步骤已执行”的校验口径。
- 统一承接逐章摘要与批量摘要模板。
- 统一承接步骤 0 初始化、工件命名、未分类兜底与连续性台账默认位置。
- 当章节创作出现卡顿、续跑失焦或返工过大时，先回到九部件与进度节奏层重建顺序，再继续正文链路，而不是盲目硬写。
- 在步骤 5–9.5 之间，统一判断这章是“真实闭环”还是“文件已齐但链路未闭”，并明确需要回写的上游层级。
- 若第 3 步去 AI 味只做了轻修但问题实为人物假、对话太顺或破局太机械，应明确标记需要升级到更高强度再回炉，不能把“做过去味”误判成“问题已解”。

## 工件命名规则（强制）

- 章节正文主文件默认不带日期；`## 作者有话说` 与 `## 章节后记` 仍附着在同一章节文件内，不单独创建带日期文件名。
- 分章终审报告与分章书评文件名必须带日期，日期格式固定为 `YYYY-MM-DD`。
- 多平台派生正文文件名默认不带日期，沿用平台目录与章节路径命名；日期只用于对应审阅报告与书评，不用于平台正文本体。

## 硬规则

- 只有当正文、终审报告、阅读笔记、分章书评、日志状态全部闭环时，才允许宣称完成。
- 本章相关人物传记、技术设定与背景设定统一视为“启动前裁判源”。
- 步骤 1–9.5 的任一写稿、润色、去味、审阅动作前，都必须先完成这组裁判源复核。
- 未完成裁判源复核时，一律视为闭环尚未真正启动，不得伪装成已进入执行阶段。
- 任一章节正文在步骤 2 之后发生改动，都必须立刻补做“基础复检包”，不得只凭旧报告或目测放行。
- “基础复检包”固定包含 3 项：①正文重跑 `scripts/count-chapter.ps1 -MinCJK 3500 -MaxCJK 6500`；②若同文件含 `## 作者有话说`，再跑 `scripts/count-afterword.ps1 -MinCJK 200 -MaxCJK 300`；③同步刷新所有受影响的审阅链。
- 字数门禁的唯一依据，是 `count-chapter.ps1` 的 `BodyCJK / Meets3500 / WithinRange`；`Len` 仅作参考，不得被“去空白统计”或任何自定义字符数替代。
- 任一日志、终审报告、读者侧摘要若出现“正文字数：XXXX（去空白统计）”但未同时给出 `BodyCJK / Meets3500 / WithinRange`，一律视为闭环未完成。
- 任一章节正文在步骤 2 之后发生改动，都必须同步刷新同文件 `## 章节后记`，至少复核：本章定位、功能点落地核对、爽点登记、伏笔与回收、设定/程序呈现、人物发展、证据要素、钩子类型、数字纪律检查、偏离记录、后续铺垫、读者吸引点与优化建议。
- 步骤 1–5 之间，任一正文改动后都必须复检正文区块是否残留“这几章 / 前几章 / 前面几章 / 前面这几章 / 后几章 / 后面几章 / 后面这几章 / 上一章 / 下一章 / 本章 / 这章 / 后文会 / 我们会看到 / 读者会发现”等章节自指 / 连载导览腔；残留即视为闭环未完成，不得进入终审通过。
- “整章 / 这整章 / 这一整章”同样属于正文区块必须清掉的章节自指 / 作者讲评腔；残留即视为闭环未完成。
- 步骤 3 / 4 / 5 / 6 / 7 / 8 / 9 不得因为“已有文件”或“分数看起来不错”被跳过。
- `9.20+` 的唯一判定，以**连续两轮独立终审报告中的综合评分在保留两位小数后均严格大于 9.20** 为准；首次过线后仍须追加一轮回炉修订与独立复审，第二轮复审也过线才允许收口。
- 命中本 Skill 时，必须坚持执行器身份，除硬性缺参外，不得把阶段汇报伪装成完成汇报。
- 每章日志必须记录主卖点保真状态，至少回答“本章怎样呈现 / 强化 / 维持卖点；若未显性呈现，下一章由谁接棒”。
- 每章日志还必须至少记录：章首抓手、中段回报、章末接棒点；若存在双轨推进，还要分别记录每条推进线新增了什么证据 / 动作 / 损失。
- 回炉判断不得只看文句顺滑度；若卖点在正文阶段变淡、变虚、变泛，也必须判为未闭环。

## 与其他 Skill / Prompt 的边界

- 本 Skill 负责编排与执行完整章节 SOP。
- 单步正文创作、审阅或专项强化，仍应交给对应的下游通用 Skill 本体处理。

## 完成检查

- 是否已读取全部必读 references
- 是否已为每章认真仔细阅读相关人物传记与技术设定 / 背景设定文档，并在日志或终审结论中写明本轮实际核对了哪些裁判源
- 是否已对每章维护日志与一致性状态
- 是否已在日志或终审结论中记录本章卖点落地度与遗忘风险
- 是否已在本轮最终稿上重跑 `scripts/count-chapter.ps1 -MinCJK 3500 -MaxCJK 6500`，并在有 `## 作者有话说` 时同步重跑 `scripts/count-afterword.ps1 -MinCJK 200 -MaxCJK 300`
- 是否已确认所有门禁判断、日志字段、终审报告和最终摘要使用的是 `BodyCJK / Meets3500 / WithinRange`，而不是“去空白统计”
- 是否已确认同文件 `## 章节后记` 已按最新模板同步更新，而不是沿用旧版字段或旧版结论
- 是否已确认正文区块不存在“这几章 / 前几章 / 前面几章 / 前面这几章 / 后几章 / 后面几章 / 后面这几章 / 上一章 / 下一章 / 本章 / 这章 / 后文会 / 我们会看到 / 读者会发现”等章节自指 / 连载导览腔
- 是否已确认正文区块不存在“整章 / 这整章 / 这一整章”这类章节自指 / 作者讲评腔
- 是否已完成步骤 10 的读者产物与步骤 11 的分章书评
- 是否已按模板输出步骤 12 摘要
