# 章节批量模式与连续性日志

## 批量模式目标

在不降低单章质量门槛的前提下，通过分相位执行减少重复工作：

- 上下文预热一次、复用多次
- 减少作者 / 审阅者 / 读者角色频繁切换
- 只对未达标章节回炉，而不是整批重来

## 批量模式输入

- `CHAPTER_FILES[]`
- `OUTLINE_FILES[]`（可选）
- `CHAPTER_BATCH[]`
- `BATCH_ID`（可选）

若提供 `CHAPTER_BATCH[]`，应以其为准；若仅提供 `CHAPTER_FILES[]`，则默认按顺序逐章建立任务表，并在缺省时复用同一个 `OUTLINE_FILE` 或对应的 `OUTLINE_FILES[]` 项。

## 批量模式执行原则

1. 先全量预检路径、编号、目录与缺参
2. 建立有序任务表
3. 分相位流水线执行
4. 逐章门槛、逐章回炉、逐章摘要
5. 失败隔离，不污染其他章节

## 分相位流水线硬约束

- 同一批次允许共享上下文预热，但不得共享“完成状态”。
- 必须按相位推进，而不是某章跑完整条链后再切下一章。
- 门槛判断与回炉必须按章节粒度执行，不得让已达标章节陪跑重做。
- 只要仍有章节未完成步骤 10 或 11，整批就只能是 `in_progress`。

## 推荐分相位顺序

1. 初始化批次
2. 批量创作与补字
3. 批量精修与去 AI 味
4. 批量审阅与回炉
5. 批量读者产物
6. 批量摘要输出

## 连续性台账

批量模式建议维护：

- `CONTINUITY_LEDGER_FILE`

最少记录：

- 章节编号锚点
- 时间线锚点
- 人物状态变化
- 能力状态 / 代价状态
- 关键证据
- 未回收钩子
- 当前章节链路最易失温对象
- 当前判断的优先回写层级

## 日志目录与续跑规则

- 每次执行都必须在 `SOP执行日志/撰写章节/` 下创建或续写日志。
- 单章日志使用 `logKey = OUTLINE_FILE + CHAPTER_FILE` 匹配最近一次未完成日志并续跑。
- 批量模式必须同时维护：
  - 批次主日志
  - 每章子日志
- `batchLogKey` 可由批次内各章的 `OUTLINE_FILE + CHAPTER_FILE` 按顺序拼接得到。

## 日志规则

### 单章子日志

- 每章一份，支持独立断点恢复
- 至少包含：`sop`、`executionId`、`logKey`、`status`、`startedAt`、`updatedAt`、`inputs`、`derived`、`checkpoint`、`steps`、`artifacts`、`errors[]`
- `checkpoint` 至少跟踪：`nextStep`、`phase`、`nextAction`
- `checkpoint` 还应尽量跟踪：`chainStatus`（closed / surface_fixed_only / needs_upstream_rewrite）与 `rewriteLevel`
- 步骤 2 需记录每轮 `iteration` 与 `Len/CJK/BodyCJK/Meets3500/WithinRange/MeetsAll`，并写明 `statsSource=scripts/count-chapter.ps1`
- 步骤 2 还需记录“节拍/轮换”同构字段：
  - `HookType`（悬念/冲突/异常/动作/对话）
  - `MidPayoff`（中段回报一句话，可回指段位）
  - `TailDriver`（章末接棒问题）
  - `RotationCheck`（与前两章是否过载重复：Yes/No）
  - `GatePass`（四拍门禁：Pass/Fail）
- 不得把“去空白统计”“NoWhitespaceLen”或任何自定义字符数写进门禁字段；如需保留参考值，只能单列为 `referenceOnly`，且不得用于达标判断
- 步骤 9.5 需记录每轮 `iteration` 与最新评分纯数字

### 批次主日志

最少记录：

- `batchId`
- `items[]`
- 每章 `status`
- 每章 `checkpoint`
- 每章 `Len/BodyCJK/Meets3500/WithinRange`
- 每章 `score`
- 每章 `consistency`
- 每章 `chainStatus`
- 每章 `rewriteLevel`
- 每章 `HookType / MidPayoff / TailDriver / RotationCheck / GatePass`

批次聚合指标建议同时记录：

- `gatePassRate`（GatePass=Pass 的章节占比）
- `rotationOverloadCount`（RotationCheck=Yes 的章节数）
- `hookTypeDistribution`（各钩子类型分布）
- `midPayoffCoverage`（MidPayoff 非空章节占比）
- `tailDriverCoverage`（TailDriver 非空章节占比）

`items[]` 中每项至少还应包含：

- `index`
- `inputs.OUTLINE_FILE / inputs.CHAPTER_FILE / inputs.RELATED_CONTEXT_FILES[]`
- `derived.REVIEW_REPORT_FILE / derived.READER_NOTE_FILE / derived.BOOK_REVIEW_FILE`
- `derived.chapterNo / derived.第X部 / derived.第Y卷`（可推导时）
- `metrics.BodyCJK / metrics.Meets3500 / metrics.WithinRange / score`
- `metrics.HookType / metrics.RotationCheck / metrics.GatePass`

## 批次门禁告警规则（新增）

- 若 `gatePassRate < 0.90`：整批不得判 `completed`，至少进入一次结构回炉。
- 若 `rotationOverloadCount >= 2`：下一批或下一轮控制卡必须强制换型钩子。
- 若 `midPayoffCoverage < 1.00`：缺失中段回报的章节一律优先回炉。
- 若 `tailDriverCoverage < 1.00`：缺失章末接棒的章节不得进入终审放行。

    "CJK": 4012,

## 失败隔离

- 某一章失败，不得污染其他章节结果
- 已启动章节可收尾到安全点
- 未启动章节可暂停，但必须在摘要中标注

## 必须落盘的更新时机

1. 步骤 0 完成后：写入派生路径与 `checkpoint.nextStep=1`
2. 每个主步骤开始时：标记 `started`
3. 每个主步骤结束时：写入 `ok=true/false` 与新的 `checkpoint`
4. 步骤 2 每次扩写和字数检测后：刷新统计字段
5. 步骤 9.5 每次提取评分与决定是否回炉时：刷新评分与 `nextAction`
6. 步骤 10 / 11 完成后：把读者笔记 / 分章书评路径写入 `artifacts`
7. 输出最终摘要前：把 `status` 写成 `completed`
8. 若硬性缺参阻塞：把 `status` 写成 `blocked_missing_input`，并在 `checkpoint.nextAction` 写清最小缺失项
