# 通用-设计事件案件引擎

一个合格的事件 / 案件引擎，不是“发生一件事”，而是同时承担局部悬疑、主线供血、规则显影与人物代价。

## 常见触发词 / 用户说法速查

- 帮我设计这个单元案
- 这个案子怎么供血主线
- 异常信号怎么做
- 证据链怎么排
- 误导怎么可证伪
- 嫌疑池怎么排才不假
- 案件结束后留什么余波

## 何时使用

- 为分卷或连续章节设计单元案 / 案件 / 事件母板。
- 审视某个事件是否真的在给主线供血，而不是只做热闹。
- 设计“异常信号 → 调查 / 对抗 → 证据链 → 误导反证 → 代价 → 余波”的完整引擎。

## 本 Skill 的核心任务

每个事件 / 案件都要至少回答：

1. 异常信号是什么。
2. 现实摩擦是什么，为什么一开始会被误判成普通问题。
3. 证据载体是什么，谁能接触到。
4. 误导链是什么，如何被证伪。
5. 嫌疑池如何变化。
6. 主线拼图推进了哪一块。
7. 代价落在谁身上。
8. 余波 / 新缺口留给后文什么压力。

## 执行顺序

1. 先读取相关卷纲 / 上游任务。
2. 用母板先写出骨架，再检查它是否供血主线。
3. 强制补足：异常信号、证据载体、误导与反证、代价、余波。
4. 若事件只提供氛围、不提供主线拼图或代价升级，则视为不合格。
5. 若涉及多人误导、多层嫌疑或跨 3–5 单元的证据投放，继续细化证据链与排序模板。

## 默认输出口径

- 默认输出一份可直接供分卷、控制卡与正文拆分调用的事件 / 案件骨架。
- 默认同时给出异常信号、证据链、误导链、嫌疑池、代价与余波。

## 最低交付

- 能指出异常信号、证据载体、误导与反证、嫌疑池变化、代价与余波。
- 能说明这起事件怎样给主线供血，而不是只做热闹单元。

## 硬规则

- 禁止只有案件表面热闹，没有主线拼图推进。
- 禁止误导不可证伪、嫌疑池假摆设或代价无落点。
- 禁止事件结束后不留下余波与后续压力。

## 与其他 Skill / Prompt 的边界

- 本 Skill 负责“事件 / 案件发动机设计”，不代替整卷规划。
- 本 Skill 产出的骨架，可直接供控制卡拆成章内场景。
- 本 Skill 不直接写正文，只负责把事件变成能写、能审、能回收的结构。
