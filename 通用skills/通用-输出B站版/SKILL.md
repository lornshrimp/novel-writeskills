# 通用-输出B站版

这是平台共性本体，负责承接“B 站版输出”的跨题材共性规则。

## 常见触发词 / 用户说法速查

- 输出B站版
- 改成 B 站能发的版本
- 做 B 站平台改写
- 帮我把这章改得更年轻读者向一点

## 何时使用

- 把任意题材章节改写成更适合 B 站读者语境的版本。
- 需要强化年轻读者可读性、讨论感、节奏清晰度与视听化场景感。
- 需要为 `题材名-输出B站版` 提供统一的平台骨架。

## 不适用情形

- 只做题材边界补充，不做平台输出本体。
- 只做多平台编排、落盘和终检，而不处理 B 站平台风格本身。

## 本层职责

- 维护 B 站平台的跨题材共性规则。
- 统一承接 B 站版输出的结构约束、标题倾向、年轻读者表达与风格门禁。
- 为后续新增题材保留直接复用入口。

## 继续读取的 references

- `references/跨题材共性与迁移来源.md`
- `references/平台共性执行细则.md`
- `references/分节级补救映射与详细规则回填.md`

## 默认执行顺序

1. 先锁定源稿不可变层：事实、事件链、证据链、角色关系与因果顺序不得改坏。
2. 读取对应题材的 `题材名-输出B站版` Skill，确认题材入口与路由关系。
3. 再读取本 Skill 的 `references/平台共性执行细则.md` 与 `references/分节级补救映射与详细规则回填.md`，执行 B站 平台的年轻读者可读性、讨论感与视听化抓力强化。
4. 由题材包装层继续读取其题材 references，锁定题材边界、题材禁行项与题材专项补丁（作为兜底约束）。
5. 在“平台优先、题材兜底”前提下执行改写正文，确保平台化表达与源稿保真同时成立。
6. 最后才处理降相似度，不得为了降重写成解说稿腔。
7. 执行 POV 契约复核，确保与 `platformPovContract` 一致，不得出现未授权人称漂移。
8. 改写输出完成后，必须立即执行“输出后自检与修订”节的五项自检，不得跳过。
9. 若自检未通过，必须按结论回炉修订并复检，直至“自检通过/自检修订完成”后方可落盘。

## POV 契约与连续性（强制）

- 若本次由 `通用-多平台输出编排` 调度，必须继承该流程已锁定的 `platformPovContract`，不得在本 Skill 内重新决定人称。
- 若用户直接命中本 Skill，则必须在改写前先从源稿正文与当前平台已完成前序章节中推断并锁定 `platformPovContract`；默认优先级为：用户明确指定 > 当前平台已完成前序章节 > 源稿既有 POV 链路 > 本批次首个已通过章节。
- 未获批准不得把连续章节从第三人称静默改成第一人称，或反向漂移；若源章本来就是视角切换章 / 多视角连续章，只能按已登记 `switchPlan` 执行，不得临场换壳。
- 落盘前后都必须显式执行 POV 校验，并把结果写入日志或执行记录；中文平台默认使用 `scripts/pov_validate.py` 或 `scripts/run_pov_gate.ps1` 以 `lang=zh` 或 `auto` 复核正文区块，第三人称链路检查对话外第一 / 第二人称，第一人称链路检查对话外第一人称锚点。

## POV 选择指南（如无显式契约）

若本次执行未获得上游明确的 `platformPovContract` 锁定，按以下指南为本平台适配稿选定**全书统一人称**：

> **人称是全书级契约**：所选人称必须贯穿本平台所有章节，不得逐章切换，不得章内混用（对话引用除外）。极特殊情况下某章须切换主视角人物时，该章开头必须明确声明「我是[角色名]」，且整章只保持该角色的第一人称叙事。

### B站版的 POV 定位

**平台特点**：视听化场景感、节奏清晰、多角色动态、年轻读者讨论文化

**默认推荐**：**第三人称（全书统一）**

B站重视视听化场景感和多角色冲突的清晰展现，第三人称便于多角色动态的「画面化」呈现；B站的弹幕讨论文化中，读者习惯评论不同角色的行为，第三人称的全景式场景比第一人称的单视角更契合这种多角色互动阅读体验。

**适合选择第三人称（全书统一）的情形**：
- 题材涉及多角色动态、群像互动或多线并行
- 需要强视觉化场景描述，视听化呈现是核心要求

**适合选择第一人称（主角第一，其他角色第三）的情形**：
- 源稿为第一人称；题材以主角个人成长/感受为绝对核心
- B站年轻读者接受第一人称的强代入感，尤其适合主角视角的沉浸式叙事

## 默认输出口径

- 默认输出一版可直接继续落盘或进入平台终检的 B 站派生正文。
- 默认保留原章核心信息、节奏清晰度与视听化场景感，不新增关键事实。

## 最低交付

- 有明确的年轻读者可读性与讨论感。
- 保住原章核心事件链、节奏推进与场景可视化。
- 读起来不像解说稿，也不像只换词不换平台感的轻改稿。

## 输出后自检与修订（必做，不得跳过）

改写输出完成后，必须立即执行本节自检；根据发现的问题就地修订，确认通过后方可落盘。**不得以任何理由跳过本步骤。**

**五项自检：**

0. **字数达标**：正文和 `## 作者有话说` 必须先满足本 Skill 的字数门禁；正文未达硬下限前，不得进入后续自检，必须先回炉补足字数。**严格禁止为凑字数进行模板化补字或机械扩写**；若检测到模板句污染、解释腔堆砌或无意义重复，则判定字数虽足但自检失败，必须重写。
1. **零新增事实**：改写版未引入源稿中不存在的关键事实、人物或设定；引入则删除/还原。
2. **核心事件链完整**：起因→推进→高潮→章末钩子方向均在改写版中体现；断裂则补回。
3. **平台风格达标**：改写版满足本 Skill“最低交付”（或平台核心风格要求）所列各项；不足则就地修订。
4. **无模板句污染**：未出现连续 2 句及以上解释腔/模板腔/口径名词句群；命中则整段回炉。
5. **POV 一致**：改写版 POV 与已锁定的 `platformPovContract` 一致；漂移则回炉修正。

**结论格式（必须输出）：**

- 全部达标：`[自检通过] 五项均达标，可落盘。`
- 已修订达标：`[自检修订完成] 已修订：<问题描述>，现五项均达标，可落盘。`
- 未通过（不得落盘）：`[自检未通过] 命中：<问题描述>，需继续处理。`

命中“未通过”后必须修订并重新输出自检结论，循环直至输出“通过”或“修订完成”方可落盘。

## 硬规则

- 绝对禁止输出“抽象自述 + 动词模板 + 口径名词”的垃圾句群；凡出现类似“对照项落在… / 先把同一句话拆碎 / 先把顺滑的解释拆开 / 只求能追溯（复核、对得上）”等模板化短语，必须判定为污染并整稿回炉。
- 严禁使用或间接调用以下脚本对正文做生成、扩写、拼接或降重：`scripts/append_cn_unique_monologue.ps1`、`scripts/append_cn_unique_narration.ps1`、`scripts/append_cn_unique_thirdperson.ps1`、`scripts/rephrase_cn_body.ps1`、`scripts/rephrase_en_body.ps1`、`scripts/cn_lexicon_profile_transform.ps1`。这些工具不得进入平台正文生产链路。
- 平台稿新增“模板句污染清零门禁”：正文不得出现连续 2 句及以上“我/我这边/我心里… + 把/将… + 对齐/校准/锁死/拆开… + 连接词 + 口径名词 + 只求/不求/先把…”结构；命中即失败，不论字数与相似度是否通过。
- 平台连续章必须服从已锁定的 `platformPovContract`；不得在本 Skill 内擅自把前序稳定链路改成人称新链路。
- 若命中未授权 POV 切换或 POV 校验失败，必须按 `pov_drift_detected` / `pov_switch_without_approval` 判定失败并整稿回炉。

- 命中本技能时，必须同时加载对应题材的 `题材名-输出B站版` Skill。
- 题材特有规则不得回写到本文件中平行维护。
- 平台派生正文默认落在本 Skill 的工作目录 `B站/` 下。
- 章节级派生正文目录统一按小说结构决定：若作品有分部，则位于 `B站/第X部/第Y卷/`；若作品无分部，则位于 `B站/第X卷/`。
- 平台派生正文文件名默认不带日期，沿用既有章节号与平台标题同步规则；日期只用于配套审阅报告、书评等派生产物。
- 章节派生正文必须以 `.md` 文件形式落盘：若对应章节目录不存在，执行前必须先创建完整目录路径，再将改写后的章节内容写入该目录下的 `.md` 文件，不得只输出聊天稿而不写入文件。
- 落盘路径的完整推断优先级：① 用户显式指定 chapterPath → ② 从源文件名与分部/卷信息自动推断 → ③ 若无法确定分部/卷，暂停并向用户确认，不得乱推断后静默落盘。
- 平台稿落盘后必须显式运行 `scripts/count-chapter.ps1 -MinCJK 3500 -MaxCJK 6500` 核正文，并运行 `scripts/count-afterword.ps1 -MinCJK 80 -MaxCJK 120` 核 `## 作者有话说`。
- 生成阶段必须先达到字数保底目标再允许落盘：正文目标 `4200–5800` CJK（硬下限 `3500`），`## 作者有话说` 目标 `80–120` CJK；未达保底目标时必须先回炉扩写，禁止"先落盘再补字数"。
- 在无损迁移完成前，不得删除源 `输出B站版.prompt.md` 中尚未完成映射的内容。

## 与其他 Skill / Prompt 的边界

- 本 Skill 只负责 B 站平台的跨题材共性骨架。
- 题材边界、题材禁行项与题材特有口径，继续由对应 `题材名-输出B站版` 承接。
