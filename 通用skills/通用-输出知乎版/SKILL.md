# 通用-输出知乎版

这是平台共性本体，负责承接“知乎版输出”的跨题材共性规则。

## 常见触发词 / 用户说法速查

- 输出知乎版
- 改成知乎能发的版本
- 做知乎盐选 / 专栏改写
- 帮我把这章改得更有逻辑链和讨论感

## 何时使用

- 把任意题材章节改写成更适合知乎读者阅读与讨论的版本。
- 需要强化逻辑可追、信息结构、观点感与节制钩子。
- 需要为题材包装层提供统一的知乎平台共性骨架。

## 不适用情形

- 只做题材边界补充，不做平台输出本体。
- 只做多平台编排、落盘和终检，而不处理知乎平台风格本身。

## 本层职责

- 维护知乎平台的跨题材共性规则。
- 统一承接知乎版输出的结构约束、标题倾向、讨论感与字数/风格门禁。
- 为 `题材名-输出知乎版` 提供稳定的通用平台骨架。
- 只处理平台共性，不替代题材包装层决定题材边界、题材禁行项、题材专项补丁与题材 refs 调度。

## 继续读取的 references

- `references/平台共性执行细则.md`
- `references/分节级补救映射与详细规则回填.md`

## 与题材包装层的协作说明

- 若对应题材已有 `题材名-输出知乎版`，必须先由题材包装层锁定：题材边界、题材禁行项、题材专项补丁、题材 refs 调度与题材输出口径。
- 本 Skill 只负责知乎平台的跨题材共性：高概念入口、逻辑链、问题意识、讨论张力、平台门禁、落盘与终检骨架。
- 若题材包装层与本 Skill 出现冲突：
	- 题材边界、题材禁行项、题材专项补丁 → 以题材包装层为准；
	- 平台共性节奏、信息结构、平台终检门禁 → 以本 Skill 为准。
- 若用户直接命中本 Skill 且仓库内存在对应题材包装层，不应绕过题材包装层直接把平台共性套到正文上。
- 若仓库内暂时不存在对应题材包装层，本 Skill 只能保守执行平台共性，不得臆造题材规则。

## 历史迁移参考（非默认调度）

- `references/跨题材共性与迁移来源.md`
- `references/平台研究补充_知乎盐选都市悬疑.md`

这些文件仅用于追溯迁移来源、核对历史承接关系或处理遗留映射缺口；默认执行时，不把它们当作当前跨题材平台规则主承载。

## 默认执行顺序

1. 先锁定源稿不可变层：事实、事件链、证据链、角色关系与因果顺序不得改坏。
2. 读取对应题材的 `题材名-输出知乎版` Skill，确认题材入口与路由关系。
3. 再读取本 Skill 的 `references/平台共性执行细则.md` 与 `references/分节级补救映射与详细规则回填.md`，执行知乎平台的高概念、现实映射、逻辑链、可验证质感与讨论张力强化。
4. 由题材包装层继续读取其题材 references，锁定题材边界、题材禁行项与题材专项补丁（作为兜底约束）。
5. 在“平台优先、题材兜底”前提下执行改写正文，确保平台化表达与源稿保真同时成立。
6. 最后才处理降相似度，不得为了降重写成泛中文平台稿。
7. 执行 POV 契约复核，确保与 `platformPovContract` 一致，不得出现未授权人称漂移。
8. 改写输出完成后，必须立即执行“输出后自检与修订”节的五项自检，不得跳过。
9. 若自检未通过，必须按结论回炉修订并复检，直至“自检通过/自检修订完成”后方可落盘。

## POV 契约与连续性（强制）

- 若本次由 `通用-多平台输出编排` 调度，必须继承该流程已锁定的 `platformPovContract`，不得在本 Skill 内重新决定人称。
- 若用户直接命中本 Skill，则必须在改写前先从源稿正文与当前平台已完成前序章节中推断并锁定 `platformPovContract`；默认优先级为：用户明确指定 > 当前平台已完成前序章节 > 源稿既有 POV 链路 > 本批次首个已通过章节。
- 未获批准不得把连续章节从第三人称静默改成第一人称，或反向漂移；若源章本来就是视角切换章 / 多视角连续章，只能按已登记 `switchPlan` 执行，不得临场换壳。
- 落盘前后都必须显式执行 POV 校验，并把结果写入日志或执行记录；中文平台默认使用 `scripts/pov_validate.py` 或 `scripts/run_pov_gate.ps1` 以 `lang=zh` 或 `auto` 复核正文区块，第三人称链路检查对话外第一 / 第二人称，第一人称链路检查对话外第一人称锚点。

## POV 选择指南（如无显式契约）

若本次执行未获得上游明确的 `platformPovContract` 锁定，按以下指南为本平台适配稿选定**全书统一人称**：

> **人称是全书级契约**：所选人称必须贯穿本平台所有章节，不得逐章切换，不得章内混用（对话引用除外）。极特殊情况下某章须切换主视角人物时，该章开头必须明确声明「我是[角色名]」，且整章只保持该角色的第一人称叙事。

### 知乎版的 POV 定位

**平台特点**：亲历叙事、逻辑链可验证、讨论张力、自述体基因

**默认推荐**：**第一人称（主角第一，其他角色第三）**

平台研究明确「第一人称或深限知质感优先」。知乎有强烈的「亲历者自述」内容基因，第一人称最契合平台的问答体/手记体/日志体入口感。即便是推理悬疑题材，「我如何推理、我如何判断」的亲历视角比「他如何推理」更能激发讨论张力——知乎读者天然愿意对「我」的判断提出质疑或支持。

**适合选择第一人称（主角第一，其他角色第三）的情形**：
- 任何可以用「我经历了什么」框架叙述的题材
- 需要引发知乎读者讨论和判断参与感的作品
- 题材强调可验证质感和现实映射

**适合选择限制第三人称（全书严格限制，禁用全知视角）的情形**：
- 源稿为成熟的第三人称链路，且已有本平台大量落盘章节，改换人称成本极高
- 若坚持第三人称，必须改为严格的限制第三人称——所有描写均通过主角感知锚点过滤，禁止全知视角

## 默认输出口径

- 默认输出一版可直接继续落盘或进入平台终检的知乎派生正文。
- 默认保留原章核心信息、逻辑链、可验证质感与讨论张力，不新增关键事实。

## 最低交付

- 有明确的逻辑可追性、讨论感与信息结构。
- 保住原章核心事件链、判断链与追读驱动。
- 读起来不像泛平台稿，也不像观点文硬套小说。


## 输出后自检与修订（必做，不得跳过）

改写输出完成后，必须立即执行本节自检；根据发现的问题就地修订，确认通过后方可落盘。**不得以任何理由跳过本步骤。**

**五项自检：**

0. **字数达标**：正文和 `## 作者有话说` 必须先满足本 Skill 的字数门禁；正文未达硬下限前，不得进入后续自检，必须先回炉补足字数。**严格禁止为凑字数进行模板化补字或机械扩写**；若检测到模板句污染、解释腔堆砌或无意义重复，则判定字数虽足但自检失败，必须重写。
1. **零新增事实**：改写版未引入源稿中不存在的关键事实、人物或设定；引入则删除/还原。
2. **核心事件链完整**：起因→推进→高潮→章末钩子方向均在改写版中体现；断裂则补回。
3. **平台风格达标**：改写版满足本 Skill“最低交付”（或平台核心风格要求）所列各项；不足则就地修订。
4. **无模板句污染**：未出现连续 2 句及以上解释腔/模板腔/口径名词句群；命中则整段回炉。
5. **POV 一致**：改写版 POV 与已锁定的 `platformPovContract` 一致；漂移则回炉修正。

**结论格式（必须输出）：**

- 全部达标：`[自检通过] 五项均达标，可落盘。`
- 已修订达标：`[自检修订完成] 已修订：<问题描述>，现五项均达标，可落盘。`
- 未通过（不得落盘）：`[自检未通过] 命中：<问题描述>，需继续处理。`

命中“未通过”后必须修订并重新输出自检结论，循环直至输出“通过”或“修订完成”方可落盘。

## 硬规则

- 绝对禁止输出“抽象自述 + 动词模板 + 口径名词”的垃圾句群；凡出现类似“对照项落在… / 先把同一句话拆碎 / 先把顺滑的解释拆开 / 只求能追溯（复核、对得上）”等模板化短语，必须判定为污染并整稿回炉。
- 严禁使用或间接调用以下脚本对正文做生成、扩写、拼接或降重：`scripts/append_cn_unique_monologue.ps1`、`scripts/append_cn_unique_narration.ps1`、`scripts/append_cn_unique_thirdperson.ps1`、`scripts/rephrase_cn_body.ps1`、`scripts/rephrase_en_body.ps1`、`scripts/cn_lexicon_profile_transform.ps1`。这些工具不得进入平台正文生产链路。
- 平台稿新增“模板句污染清零门禁”：正文不得出现连续 2 句及以上“我/我这边/我心里… + 把/将… + 对齐/校准/锁死/拆开… + 连接词 + 口径名词 + 只求/不求/先把…”结构；命中即失败，不论字数与相似度是否通过。
- 平台连续章必须服从已锁定的 `platformPovContract`；不得在本 Skill 内擅自把前序稳定链路改成人称新链路。
- 若命中未授权 POV 切换或 POV 校验失败，必须按 `pov_drift_detected` / `pov_switch_without_approval` 判定失败并整稿回炉。

### 叙述人称改写规范（知乎平台专属）

- 命中 `通用-多平台输出编排` 或当前平台已存在前序章节时，必须先继承已锁定的 `platformPovContract`；知乎版不得自行重选人称。
- 一般情况下，改写知乎版时应尽量保持原作叙述人称不变，避免随意切换人称。
- 只有 `switchApproved=true` 且已写明 `switchReason / switchPlan` 时，才允许把稳定第三人称链路切成第一人称，或改由非主角第一人称叙述。
- 如需将章节改写为第一人称，首选固定小说第一主角为第一人称，其它角色均为第三人称。
- 若确有需要以非主角的第一人称视角改写某章节，必须在章节开篇明确说明“我是谁”，避免读者混淆。
- 无论何种情况，一个章节中最多只能有一个角色以第一人称视角出现，不得多角色并行第一人称。
- 改写完成后必须用 `scripts/pov_validate.py` 或 `scripts/run_pov_gate.ps1` 做正文区块 POV 复核；若与已锁定契约不一致，一律回炉。
- 题材包装层如有更细致的人称规则，优先遵循题材包装层，平台共性规则为兜底约束。

- 命中本技能时，必须同时加载对应题材的 `题材名-输出知乎版` Skill。
- 题材特有规则不得回写到本文件中平行维护。
- 修改知乎平台共性规则时，应优先修改本 Skill，而不是多个题材入口。
- 不得把历史迁移参考里的 `都市悬疑` 口径直接当成跨题材默认规则继续扩散。
- 平台派生正文默认落在本 Skill 的工作目录 `知乎/` 下。
- 知乎编辑器的小节标题字数上限按 **15 字** 执行；章节派生与分发时必须以可保存为最高优先级，标题不得超过 15 字。
- 若原始章节标题超过 15 字，必须先压缩为平台合规标题再保存；卷号/章号信息通过文件名与分发记录保持一致，不强制写入站内标题。
- 章节级派生正文目录统一按小说结构决定：若作品有分部，则位于 `知乎/第X部/第Y卷/`；若作品无分部，则位于 `知乎/第X卷/`。
- 平台派生正文文件名默认不带日期，沿用既有章节号与平台标题同步规则；日期只用于配套审阅报告、书评等派生产物。
- 章节派生正文必须以 `.md` 文件形式落盘：若对应章节目录不存在，执行前必须先创建完整目录路径，再将改写后的章节内容写入该目录下的 `.md` 文件，不得只输出聊天稿而不写入文件。
- 落盘路径的完整推断优先级：① 用户显式指定 chapterPath → ② 从源文件名与分部/卷信息自动推断 → ③ 若无法确定分部/卷，暂停并向用户确认，不得乱推断后静默落盘。
- 平台稿落盘后必须显式运行 `scripts/count-chapter.ps1 -MinCJK 3500 -MaxCJK 6500` 核正文，并运行 `scripts/count-afterword.ps1 -MinCJK 200 -MaxCJK 300` 核 `## 作者有话说`。
- 生成阶段必须先达到字数保底目标再允许落盘：正文目标 `4200–5800` CJK（硬下限 `3500`），`## 作者有话说` 目标 `220–280` CJK（硬下限 `200`）；未达保底目标时必须先回炉扩写，禁止“先落盘再补字数”。
- 在无损迁移完成前，不得删除源 `输出知乎版.prompt.md` 中尚未完成映射的内容。

## 与其他 Skill / Prompt 的边界

- 本 Skill 只负责知乎平台的跨题材共性骨架。
- 题材边界、题材禁行项与题材特有口径，继续由对应 `题材名-输出知乎版` 承接。
- 题材 Prompt 只应路由到 `题材名-输出知乎版`；不应直接把本 Skill 与题材 refs 并列成双入口。
