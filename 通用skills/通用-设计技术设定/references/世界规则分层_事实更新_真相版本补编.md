# 世界规则分层_事实更新_真相版本补编

> 吸收外部 `planner-skill` 中对世界观文档与归档层的分工做法，用于把技术设定与规则文档写成“可分层、可更新、可区分真相版本”的作者侧体系。

## 一、规则文档分层建议

若当前项目的设定量较大，默认可拆成：

- 地理 / 地点
- 世界规则 / 关键机制
- 组织 / 势力
- 时间线
- 专项技术设定 / 手法文档

不必机械照搬目录，但至少要让：

- 空间信息
- 规则信息
- 势力信息
- 时间信息

能被清楚区分与回指。

## 一点五、来源层速判（新增）

- `规则事实层`：规则、边界、代价、已发生变化与专项设定文档分层是否清楚。
- `真相版本层`：作者真相、角色已知版本、误解 / 传闻 / 官方口径是否分开维护。
- `修订归档层`：何时更新、为何更新、按正文还是旧设定为准、如何写修订说明是否明确。

## 二、只记录已确立事实

技术设定文档默认记录的是：

- 已经建立或确认的规则
- 已经显影的边界和代价
- 已经存在的版本差异
- 已经发生的改变

不要把“未来可能会这样写”误塞进事实设定层。

## 三、作者真相 vs 角色已知版本

涉及历史、神话、制度、技术真相时，最好尽量区分：

- **作者知道的真相**：真正成立的底层机制
- **角色知道的版本**：角色当前认知、误解、传闻、官方口径

写正文时，角色只能调用自己知道的版本；不能替作者剧透底层真相。

## 四、状态更新场景

以下情况应触发文档补写或修订：

- 新规则首次被建立或揭示
- 旧规则出现例外、反制或失效条件
- 新势力或新地点对设定产生结构性影响
- 某条技术路径进入新阶段
- 某个关键误导被证伪，真相版本切换

## 五、与章节定稿的关系

章节定稿后，如正文实际呈现与旧设定冲突：

- 先确认是正文写歪，还是设定应被迭代
- 若最终以正文为准更新设定，需保留修订说明
- 涉及关键规则边界变化时，最好记录本轮修订时间与对齐口径

## 六、设定文档的最低可调用性

每份关键规则文档最好都能快速回答：

- 这条规则是什么
- 谁能调用 / 触发
- 触发条件是什么
- 会留下什么痕迹
- 代价是什么
- 最容易露馅的破绽是什么
- 不能直接解决什么问题

## 七、红线

- 不得把未来规划当成既定事实
- 不得让角色版本和作者真相长期混写不分
- 不得只有酷炫概念，没有阶段、限制、代价与证据痕迹

## 八、最小版本工件（新增）

```markdown
【规则事实层】当前已确认规则 / 边界 / 代价 / 文档分层
【真相版本层】作者真相 / 角色已知版本 / 当前误解
【修订归档层】更新触发 / 采用口径 / 本轮修订说明
【当前判定】Green / Yellow / Red
```

## 九、Green / Yellow / Red 版本判定（新增）

- `Green`：三层都清楚，既能防吃书，也能避免角色替作者剧透。
- `Yellow`：规则事实层已立，但版本区分或修订归档仍偏弱，长线维护有风险。
- `Red`：事实、版本、未来规划混写，后续越写越容易双口径打架。
