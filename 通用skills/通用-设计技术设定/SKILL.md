# 通用-设计技术设定

把技术设定写成作者能调用的“规则 + 痕迹 + 代价”文档，而不是技术说明书。

## 保真迁移要求（强制）

命中技术设定设计任务时，除本文件外，**必须继续读取**：

- `references/技术设定模板与镜头包.md`
- `references/证据链与程序摩擦规则.md`
- `references/创新性要素库与阶段红线.md`
- `references/中国语境与一致性检查.md`
- `references/结构规范与真实感要求.md`
- `references/长篇演进与城市语境.md`
- `references/科学对冲玄学与能力代价设计.md`
- `references/示例格式与验证流程.md`
- `references/现实职业绑定与前三章验证.md`

## 常见触发词 / 用户说法速查

- 帮我做技术设定
- 重写这套规则文档
- 这条手法边界不清
- 给我补证据链和程序摩擦
- 这份设定太像讲义了

## 何时使用

- 新建某个技术设定 / 手法 / 程序链 / 证据载体文档。
- 重写已有规则文档。
- 给设定补“可证据化映射、镜头包、误导与证伪对”。
- 修复设定万能化、讲课化、空概念化问题。

## 不适用情形

- 只做设定审阅，不改稿：优先使用 `通用-审阅技术设定`。
- 只想在聊天里讨论概念，不落作者侧资料文件。

## 默认输出口径

- 默认产物是**作者侧资料文档**：供大纲、正文、台账交叉引用与核查，不是读者向设定科普。
- 默认直接创建或修改真实资料 Markdown 文件，而不是只在聊天里脑暴。
- 当用户打开某个技术设定文件并调用本技能时，必须把生成或修订后的“手法机制 / 程序链条 / 证据载体与字段 / 现实语境与限制 / 在主线与单元事件中的作用”写入目标文件的对应小节。
- 对本轮涉及的每个关键要素，至少补齐：**一句话口径（动作→凭证→后果）+ 误导点与证伪路径 + 成本如何在正文里看得见**。
- 若用户明确要求“完全重写”，按模板骨架覆盖重写；若只要求补某一块，则在对应小节追加或替换，默认保留其他未涉及部分。
- 若用户要求为新技术概念单独建档，默认在对应资料目录下新建 `要素名.md`，并确保其时间线与长篇演进口径一致。
- 若原文件历史较长且用户要求“保留原文、只补最新版本”，则在文末追加“本轮修订（YYYY-MM-DD）”小节；若用户未说明，默认采用保守补充和纠偏策略，不直接删除原有设定。
- 聊天侧必须简要说明：写入或新建了哪个资料文件、采用覆盖还是追加策略、是否已对齐总纲 / 多面体 / 伏笔台账等上游口径。

## 设计目标

1. 规则必须能拆成条款。
2. 每条关键规则都尽量有痕迹可取。
3. 设定必须有边界、代价、漏洞与反制。
4. 必须给正文提供镜头化写法。
5. 必须能对齐长篇阶段演进，而不是一次性用完。
6. 必须服务作者调用体验，而不是写成只适合阅读的说明书。
7. 高价值设定默认尽量压成“动作 → 载体 / 字段 → 时间 / 版本 → 后果”的调用格式，减少空概念漂浮。

## 默认执行顺序

1. 确认目标设定文件、修改范围与写回策略。
2. 先用模板与镜头包、示例格式搭骨架，并锁定作者向资料口径。
3. 再用证据链与程序摩擦规则补边界、代价、误导与证伪路径。
4. 再用创新性要素库、长篇演进与城市语境对齐阶段使用策略。
5. 涉及超常、民俗或能力机制时，再补读科学对冲玄学与能力代价设计。
6. 最后用中国语境与一致性检查、结构规范与真实感要求和验证流程收口，并实际写回文件。

## 最低交付

- 能写清规则条款、边界、代价、漏洞与反制。
- 能给出证据载体、程序摩擦与镜头化写法。
- 能作为作者侧资料文件直接被总纲、正文与台账调用。

## 硬规则

- 禁止万能设定。
- 禁止只有术语没有动作链。
- 禁止只有设定酷炫没有现实摩擦。
- 每个高价值设定都必须写“最容易露馅的破绽”。
- 除非用户明确要求“先讨论、不写文件”，否则禁止只在聊天中输出长篇设定而不落库。
- 涉及现实高风险可复刻细节时，必须叙事化、留白化、降教学性。
- 高价值设定必须能回答：它如何在前三章留下痕迹、谁先为它付代价、主角如何第一次验证它。
- 高价值设定默认优先给出原始载体、时间戳 / 版本号、操作人 / 权限与回执 / 日志出口，而不是只给概念名词。

## 与其他 Skill / Prompt 的边界

- 本 Skill 负责技术设定本体的设计与重写。
- 若用户只要体检、评分和回写建议，应优先使用 `通用-审阅技术设定`。
